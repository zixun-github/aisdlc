# AI SDLC 使用介绍 Slides 内容稿

> 用途：直接用于制作 slides 的逐页内容稿。  
> 受众：公司内部产研团队、技术负责人、项目负责人。  
> 建议页数：15 页。  
> 主线：`AI Coding 发展 -> 本项目方法与进展 -> 工具适用场景 -> 下一步推广`

## 整体风格建议

- 风格定位：内部专业汇报，简洁、克制、结论先行
- 视觉建议：深色标题 + 浅底，适量用蓝色/青色强调关键术语
- 图表优先：时间线、对比表、结构图、流程图、三段式卡片
- 文案原则：每页 3-5 个要点，避免大段长文

## 第 2 页：为什么现在要做这件事

**标题**  
为什么现在必须升级 AI 使用方式

**页面内容**  
- AI 已经从“生成代码片段”走向“参与完整研发流程”
- 单靠 prompt 已难支撑真实项目中的协作、约束与复用
- 团队普遍面临知识分散、上下文丢失、AI 输出不稳定的问题
- 真正的分水岭，不是会不会用 AI，而是有没有让 AI 稳定工作的工程体系

**页面结论**  
我们需要从“使用 AI 工具”，升级为“建设 AI 可消费的工程体系”

**布局风格**  
- 左右分栏
- 左侧放行业趋势，右侧放内部痛点
- 顶部放一句结论型大字，底部用一行总结收口

## 第 3 页：AI Coding 发展全景

**标题**  
AI Coding 的 5 个演进阶段

**页面内容**  
1. Vibe Coding：凭感觉提需求，让 AI 直接生成代码
2. Prompt Engineering：通过更好的提问方式提升输出质量
3. Context Engineering：让 AI 看到正确的项目上下文
4. Spec-Driven Development：让 AI 基于结构化规格工作
5. Harness Engineering：用系统机制约束、验证、纠偏 AI

**页面结论**  
AI Coding 的本质演进，是从“人适应 AI”走向“系统驾驭 AI”

**布局风格**  
- 横向时间线
- 每个阶段一个卡片，包含“阶段名 + 一句话定义”
- 上方是年份，下方是可靠性逐步上升的箭头

## 第 4 页：前 3 个阶段解决了什么，为什么还不够

**标题**  
从会写代码，到能在项目里工作

**页面内容**  
| 阶段 | 主要进步 | 核心局限 |
|---|---|---|
| Vibe Coding | 快速生成代码片段 | 没有项目上下文，靠人工修补 |
| Prompt Engineering | 输出质量更稳定 | 依然依赖人手工组织输入 |
| Context Engineering | 更像“在真实项目中写代码” | 仍缺少流程门禁、验证和追溯 |

**页面结论**  
前 3 个阶段解决了“写得快、写得像”，但还没有解决“做得对、做得稳”

**布局风格**  
- 表格型页面
- 上方一句结论，主体放 3 行对比表
- 右下角可单独高亮一句“缺口：缺少流程约束与质量闭环”

## 第 5 页：为什么会走到 Spec-Driven Development

**标题**  
为什么 AI 需要结构化规格，而不是一句话需求

**页面内容**  
- 当 AI 只写代码时，一句需求可能还能工作
- 当 AI 参与需求、设计、计划、测试时，口头描述就不够了
- 必须给 AI 一条结构化输入链，让每一步有明确输入和输出
- Spec as Code 的价值在于：可评审、可验收、可追溯、可复用

**建议展示链路**  
`raw -> solution -> prd -> design -> plan -> execute -> verification`

**页面结论**  
Spec-Driven Development 让 AI 从“代码生成器”变成“研发流程参与者”

**布局风格**  
- 中轴流程图
- 中间放阶段链路，左右补充“为什么需要”和“带来的价值”

---

## 第 6 页：2026 的关键跃迁是 Harness Engineering

**标题**  
2026：从 Prompt 到 Harness

**页面内容**  
**Harness Engineering 的核心判断**  
模型不是瓶颈，围绕模型的系统才是

**四大支柱**  
- Inform：Agent 能看到什么
- Constrain：Agent 能做什么
- Verify：Agent 做对了吗
- Correct：Agent 做错了怎么办

**页面结论**  
Harness Engineering 不是替代 Prompt、Context、Spec，而是把它们工程化、机械化、闭环化

**布局风格**  
- 四卡片布局
- 中间放“Harness Engineering”主标题，四周放四个支柱
- 每个支柱用一句问句解释

## 第 8 页：核心设计一，双层 SSOT

**标题**  
核心设计：项目级 SSOT + 需求级 Spec Pack

**页面内容**  
- 项目级 SSOT：长期、稳定、可治理的项目知识资产
- 需求级 Spec Pack：围绕单需求的交付闭环与执行证据
- 两层分离的原因：避免长期知识和短期交付信息混杂
- 需求完成后，通过 Merge-back 把可复用资产晋升回项目级

**页面结论**  
双层 SSOT 解决的是“既要长期沉淀，又要支持单需求快速闭环”的矛盾

**布局风格**  
- 上下双层结构图
- 上层是 `project/`，下层是 `specs/<feature>/`
- 中间用箭头表示“按需读取”和“Merge-back 晋升”

## 第 9 页：核心设计二，渐进式披露 + Router/Worker

**标题**  
核心机制：先看地图，再按需取证

**页面内容**  
**渐进式披露**  
- AI 默认先读取项目级地图与规范
- 只有任务明确指向某个需求时，才读取对应 Spec Pack
- 价值是减少上下文噪音、避免跨需求污染

**Router / Worker 分离**  
- Router 负责决定下一步做什么
- Worker 负责本阶段落盘、自检、回写
- 价值是避免技能各自分流、步骤混乱和口径漂移

**页面结论**  
这套机制的核心作用是减少脑补、避免跳步骤、降低上下文漂移

**布局风格**  
- 左右双栏
- 左边画“读取顺序”，右边画“职责分离”
- 中间放一句总括结论

## 第 14 页：使用场景二，存量项目逆向、重构与效能工具

**标题**  
使用场景 2：存量治理与工程提效

**页面内容**  
**场景 A：项目逆向 Discover**  
- 把已有代码、结构、模块关系沉淀成 AI 可消费的知识资产

**场景 B：重构场景**  
- 先冻结不变量和基线，再进入设计与实现，降低重构失控风险

**场景 C：效能工具或小需求**  
- 可以走更短闭环，但仍保持最小门禁与追溯要求

**页面结论**  
这套工具不是只适合“新项目”，同样适合存量项目治理和小需求提效

**布局风格**
