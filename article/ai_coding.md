# AI Coding 演进简史：从对话到工程

> 面向产研团队的快速科普 | 2026-03

---

## 为什么产研团队需要关注这个话题？

2024 至 2026 年，AI 辅助编码经历了几次范式跃迁。每一次跃迁，不只是"工具更强了"，而是**人与 AI 的协作模式发生了根本变化**——产品经理需要提供什么输入、研发需要建什么基础设施、QA 需要关注什么风险，都在跟着变。

理解这条演进线，有助于团队在"用 AI 提效"时少走弯路，把精力花在真正产生杠杆的地方。

---

## 一、Vibe Coding：凭感觉写代码（2024）

**核心特征**：打开 AI 对话框，用自然语言描述想要什么，AI 生成代码，人 copy-paste 到项目里。

```
"帮我写一个用户登录页面，用 React + Tailwind"
→ AI 输出一段代码
→ 开发者粘贴、跑起来、手动调整
```

这个阶段像"结对编程中那个打字特别快的搭档"——它能快速产出代码片段，但：

- **没有项目上下文**：AI 不知道你的项目结构、技术栈约定、已有组件
- **没有质量保障**：生成的代码能不能跑、风格是否一致、有没有安全问题，全靠人肉检查
- **不可复现**：同样的问题，换个对话窗口可能得到完全不同的答案

**适用场景**：原型验证、学习新技术、写一次性脚本。

**局限**：一旦项目规模超过几百行，Vibe Coding 的效率就急剧下降——AI 反复写出跟现有代码风格不一致、跟已有模块重复的代码，人要花更多时间修补。

---

## 二、Prompt Engineering：学会提问（2024—2025）

**核心特征**：人们发现 AI 的输出质量高度依赖输入质量，于是开始系统化地研究"怎么问"。

关键实践包括：

- **角色设定**：告诉 AI 它是"一个资深 React 开发者"
- **Few-shot 示例**：给 AI 看几个期望的输入输出样本
- **思维链（Chain-of-Thought）**：要求 AI 先分析再动手
- **结构化输出**：指定 JSON Schema、Markdown 模板等输出格式

```
你是一名资深后端工程师。请基于以下 API 契约，生成 Go 的 handler 代码。
要求：使用 gin 框架、标准错误处理、单元测试。
契约：[粘贴 OpenAPI 片段]
```

**进步**：输出质量显著提升，开始有了"可重复的方法论"。

**局限**：Prompt 能携带的信息量有限。对于真实项目，"怎么问"远不如"AI 能看到什么"重要。

---

## 三、Context Engineering：让 AI 看到正确的信息（2025）

**核心特征**：人们意识到，**模型能力不是瓶颈，喂给模型的上下文才是**。从"怎么问"转向"让 AI 看到什么"。

这个阶段的标志性实践：

- **AGENTS.md / Rules**：在仓库里放置项目级指令文件，让 AI 自动加载项目约定
- **RAG（检索增强生成）**：根据当前任务，动态检索相关代码和文档喂给模型
- **知识库构建**：把架构文档、API 契约、业务规则结构化地组织起来，供 AI 按需消费
- **长上下文窗口**：模型上下文从 4K→128K→1M token，能"看"更多内容

```
[自动注入] 项目技术栈：Go + gin + PostgreSQL
[自动注入] 编码规范：error wrapping、structured logging
[自动注入] 相关模块代码：user/service.go, user/repository.go
[用户指令] 在 user 模块新增"修改密码"功能
```

**进步**：AI 开始能写出"像项目里已有代码"的代码，而不是通用示例。

**局限**：即使 AI 看到了正确的信息，它仍然可能做出错误的决策——因为没有人约束它的行为边界，也没有机制验证它的输出。

---

## 四、Spec-Driven Development：规范驱动（2025）

**核心特征**：给 AI 的不再是"一句话需求"，而是**结构化的规范文档**——PRD、设计文档、API 契约、验收标准，形成完整的输入链。

核心理念：

- **Spec as Code**：需求文档像代码一样纳入版本控制、走 Review、可追溯
- **双层 SSOT**：项目级知识（架构、契约）+ 需求级 Spec Pack（从 PRD 到发布的闭环产物）
- **阶段化推进**：需求澄清 → 方案设计 → 实施计划 → 编码执行 → 测试验证，每个阶段有明确的输入/输出/门禁
- **AI 参与全流程**：不仅是写代码，还参与需求分析、方案设计、测试用例生成

```
[R1 需求澄清] raw.md → solution.md
[R2 PRD 编写] solution.md → prd.md（含 AC）
[D1 技术调研] prd.md + 项目知识库 → research.md
[D2 方案设计] research.md → design.md（含 Impact Analysis）
[I1 实施计划] design.md → plan.md（任务清单）
[I2 编码执行] plan.md → 代码实现 + 测试
```

**进步**：AI 从"代码片段生成器"变成"SDLC 全流程参与者"。产出物可追溯、可评审、可验收。

**局限**：所有的流程约束和门禁都是通过"告诉 AI 应该怎么做"来实现的——本质上是建议，不是强制。模型可能跳过步骤、猜测路径、一次性糅合多个阶段。

---

## 五、Harness Engineering：驾驭工程（2026）

**核心命题**：**模型不是瓶颈，围绕模型的系统才是。**

Harness（驾驭框架）指的是**包裹在 AI Agent 周围的整套控制系统**——不是改模型，而是改模型运行的环境。

> LangChain 仅改 Harness（不换模型）就将 Terminal Bench 排名从 Top 30 跳到 Top 5。

### 四大支柱

| 支柱 | 核心问题 | 典型手段 |
|------|---------|---------|
| **Inform（告知）** | Agent 能看到什么？ | 仓库即 SSOT、动态上下文映射、渐进式披露 |
| **Constrain（约束）** | Agent 能做什么？ | 架构边界、工具白名单、Router/Worker 分离 |
| **Verify（验证）** | Agent 做对了吗？ | 自验证循环、CI 门禁、Pre-completion Checklist |
| **Correct（纠偏）** | Agent 做错了怎么办？ | Loop Detection、自修复、经验回流 |

关键区别在于：之前所有阶段的约束都是**"告诉 AI 要怎么做"（Prompt 级）**，而 Harness Engineering 要求**"在代码和系统层面强制执行"（机械强制）**。

```
# 之前：Prompt 级约束
"你必须先执行 spec-context，获取 FEATURE_DIR 后才能读写文件"
→ AI 可能在压力下跳过

# 现在：Harness 级约束
spec-context 脚本自动执行 → 返回 FEATURE_DIR → 校验脚本验证产物结构
→ 不通过则流程物理阻断，AI 无法继续
```

### 与前几个阶段的关系

Harness Engineering 不是对前几个阶段的否定，而是**集大成**：

| 演进阶段 | 在 Harness 中的位置 |
|---------|-------------------|
| Prompt Engineering | Inform 支柱的一部分（如何组织指令） |
| Context Engineering | Inform 支柱的核心（如何组织上下文） |
| Spec-Driven Development | Inform + Constrain 支柱（结构化输入 + 阶段门禁） |
| **Harness Engineering** | **四大支柱的完整闭环（Inform + Constrain + Verify + Correct）** |

## 七、对产研团队的实操启示

### 产品经理 / 业务分析师

| 演进阶段 | 你需要做什么 |
|---------|------------|
| Vibe Coding | 口头说需求，等开发交付 |
| SDD 阶段 | 写清楚验收标准（AC）、业务规则、影响分析 |
| Harness 阶段 | 参与知识库建设——你写的 PRD/AC 就是 Agent 的输入，质量直接决定 AI 产出质量 |

### 研发工程师

| 演进阶段 | 你需要做什么 |
|---------|------------|
| Vibe Coding | 问 AI → 粘贴代码 → 手动调整 |
| Context Eng. | 维护 AGENTS.md、代码规范文件，让 AI 了解项目 |
| Harness 阶段 | 建设约束基础设施（校验脚本、CI 门禁、产物结构检查），让 AI 在"安全围栏"内工作 |

### QA / 测试

| 演进阶段 | 你需要做什么 |
|---------|------------|
| Vibe Coding | 测试 AI 生成的代码（跟人写的没区别） |
| SDD 阶段 | 基于 Spec 生成测试用例，确保 AC 可追溯 |
| Harness 阶段 | 参与 Verify 支柱建设——定义"AI 做对了"的标准，设计自验证规则 |

## 八、常见误区

**误区 1："我们直接上 Harness Engineering 就行"**

每个阶段都是下一个阶段的基础。如果团队连结构化的 Spec 输入都没有（SDD 阶段），直接做 Harness 只会得到"精心约束的垃圾输入→精心验证的垃圾输出"。

**误区 2："这是研发的事，跟产品无关"**

AI Agent 的输出质量上限取决于输入质量。PRD 写得含糊、AC 缺失、业务规则靠口头传递——再好的 Harness 也救不了。

**误区 3："用了 AI 就不需要 Review 了"**

恰恰相反。AI 能快速产出大量代码，但"快速产出的错误代码"比"慢慢写的错误代码"危害更大。Harness Engineering 的 Verify 支柱存在的意义就是：**信任但验证**。

**误区 4："Harness 是固定不变的"**

好的 Harness 应该是"可撕除的"（Rippable）——随着模型能力提升，之前需要靠 Harness 约束的某些能力可以逐步移除。过度约束会带来不必要的复杂度和成本。

---

## 总结

AI Coding 的演进本质上是一条**从"人适应 AI"到"系统驾驭 AI"**的路径：

1. **Vibe Coding**：人去适应 AI 的输出
2. **Prompt Engineering**：人学习如何更好地向 AI 提问
3. **Context Engineering**：人为 AI 组织更好的输入信息
4. **Spec-Driven Development**：人为 AI 建立结构化的工作流程
5. **Harness Engineering**：人为 AI 建立机械强制的控制系统

每一步都在减少对"AI 自律"的依赖，增加"系统保障"的比重。**越成熟的团队，越少依赖模型自身的聪明，越多依赖围绕模型的工程体系。**

这也是为什么我们在推进 AI SDLC 项目知识库——它不只是"一堆文档"，而是 Harness Engineering 在 SDLC 领域的落地实践：用结构化的知识告知 Agent、用阶段门禁约束 Agent、用产物校验验证 Agent、用反馈回路纠偏 Agent。
