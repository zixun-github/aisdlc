> 目的：为 D2 的决策文档补足“现状/约束/风险/未知项”的**可复用结论**，并把所有不确定性收敛为**可执行验证清单**（含 Owner/截止/动作），使 D2 可以直接引用而无需重复解释。
>
> 原则：结论先行；只保留支撑决策的最小信息；不写“待确认问题/TODO”清单——未知统一进入“验证清单”。
>
> 落盘位置：`{FEATURE_DIR}/design/research.md`

## 0. 基本信息

- 需求标识（分支 / ID）：【必填】
- 作者 / 参与评审：【必填】
- 状态：draft / reviewing / approved
- 最后更新：YYYY-MM-DD
- 关联链接（讨论、资料、工单）：【可选】

## 1. 结论摘要（先给结论，3–7 行）

- 现状（1 句）：【必填】
- 最大风险（1 句）：【必填】
- 推荐方向（机制级 1 句）：【必填】
- 优先验证的 1–3 个点（引用“验证清单”编号）：【必填，例如 R1/R2/R3】

## 2. 现状与问题域（最小信息）

- 关键现状（1–5 条）：【必填】
- 痛点与影响（1–5 条）：【必填】

## 3. 范围边界与不变量（对齐 `requirements/solution.md`）

- In（明确做什么）：【必填】
- Out（明确不做什么）：【必填】
- 不变量（不会改变的语义/口径/安全边界）：【必填】

## 4. 关键约束（只写与决策相关的）

- 合规/安全：【可选】
- 性能/容量：【可选】
- 依赖/组织：【可选】
- 兼容性/迁移：【可选】

## 5. 风险与验证清单（必填，可执行）

> 要求：每条不确定性都要落到“验证信号 + 方法 + Owner + 截止 + 触发动作”。
> 不允许出现“待确认问题清单 / TODO”。

|---|---|---|---|---|---|---|
| R1 |  |  |  |  |  |  |
| R2 |  |  |  |  |  |  |

## 6. 备选与权衡（可选，2–3 个）

- 备选 A：一句话核心差异 + 适用前提 + 不选原因
- 备选 B：一句话核心差异 + 适用前提 + 不选原因

## 7. 证据入口（追溯链接）

- `{FEATURE_DIR}/requirements/solution.md` 引用点位（段落/条目/链接）：【必填】
- 相关 `project/contracts/` / `project/adr/` 入口（如适用）：【可选】
- 若缺少证据：必须在“验证清单”中补齐对应验证项：【必填】

## 8. 迭代记录（追加，不覆盖）

- YYYY-MM-DD：本轮变更摘要（相对上一轮改了什么、为什么）【必填】

## 9. 自检清单（D1-DoD）

- [ ] 不出现“待确认问题/TODO”清单；未知统一进入“验证清单”（Owner/截止/动作明确）
- [ ] TL;DR ≤ 7 行，且包含现状、最大风险、推荐方向与优先验证点（引用 R#）
- [ ] 结论可追溯（至少一个证据入口；缺失则进入验证清单）
- [ ] D2 可直接引用（无需重复解释即可用作决策输入）
