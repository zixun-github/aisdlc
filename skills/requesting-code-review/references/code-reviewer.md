## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否得当？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 边界情况是否处理？

**架构：**
- 设计决策是否合理？
- 可扩展性考虑？
- 性能影响？
- 安全顾虑？

**测试：**
- 测试是否真正测试逻辑（而非 mock）？
- 边界情况是否覆盖？
- 是否需要集成测试？
- 所有测试是否通过？

**需求：**
- 是否满足计划中所有需求？
- 实现是否匹配规格？
- 无范围蔓延？
- 破坏性变更是否文档化？

**生产就绪度：**
- 迁移策略（若有 schema 变更）？
- 是否考虑向后兼容？
- 文档是否完整？
- 无明显 bug？

### 评估

**可以合并？** [Yes/No/With fixes]

## 关键规则

**要做：**
- 按实际严重程度分类（并非全是 Critical）
- 具体（file:line，不模糊）
- 说明问题为何重要
- 承认优点
- 给出明确结论

**不要：**
- 未检查就说「看起来不错」
- 把小事标为 Critical
- 对未审查的代码给反馈
- 模糊（「改进错误处理」）
- 回避给出明确结论
