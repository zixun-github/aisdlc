## 0. 目标与定位

本设计文档用于细化 Spec 级需求分析中的 R8「前端 DEMO 页面生成（可选）」：在 R7 产出 `{FEATURE_DIR}/requirements/prototype.md`（线框原型/文本原型说明）后，把线框原型尽快落成**可运行、可走查**的前端 DEMO 页面，以便：

- 让 PM/UX/DEV/QA 以“可点击/可浏览”的方式走读核心场景链路
- 在进入生产实现前尽早发现：页面缺口、跳转不闭环、关键文案/控件缺失、状态覆盖不足等问题
- 把走查结果回流到上游文档（R6/R7 以及必要时 R1/R3/R5），减少后续返工

> 定位约束：R8 产物是 **DEMO（非生产实现）**。它不负责最终视觉、组件沉淀、接口对接、状态管理选型等“工程化实现细节”；这些应下沉到后续研发与详细设计。

## 1. 术语与接口（命令必须遵守）

### 1.1 上下文与路径

- **FEATURE_DIR**：由 `Get-SpecContext` 自动获取，定位到 `.aisdlc/specs/{num}-{short-name}/`
- **必读输入**：
  - `{FEATURE_DIR}/requirements/prototype.md`
- **可选输入**（用于补齐路由/入口与场景链路一致性）：
  - `{FEATURE_DIR}/requirements/interaction.md`
  - `{FEATURE_DIR}/requirements/prd.md`
  - `project/memory/structure.md`（若存在：用于自动获取/推断 DEMO 工程路径）
  - `project/memory/glossary.md`
  - 既有 `{FEATURE_DIR}/implementation/demo.md`（若存在：用于增量更新/对照）

### 1.2 用户必填输入（门禁参数）

- **前端 DEMO 项目根路径**：一个可访问的本地路径，指向可运行的前端 DEMO 工程根目录。

获取规则（必须遵守）：

1) **优先自动获取**：若存在 `project/memory/structure.md`，R8 必须先尝试从中提取/推断“前端 DEMO 项目根路径”。  
2) **自动获取失败再提问**：若无法从 `project/memory/structure.md` 获取到“明确且可用”的路径，R8 必须提示用户手动指定该路径（作为门禁参数）。

建议的 `structure.md` 记录方式（用于降低歧义，推荐但不强制）：

- 以“单行键值”显式写出，例如：
  - `前端 DEMO 项目根路径：E:\path\to\demo`
  - `DEMO_ROOT=E:\path\to\demo`
  - `demo_root: E:\path\to\demo`
- 若出现多个候选路径：以“存在且可访问、并能识别 `package.json`（或等价前端工程特征）”者为准；仍无法确定则进入提问。

约束：

- 必须是目录，且目录中应能识别到前端工程的基本特征（例如 `package.json` / 常见工程结构）。
- 若无法识别为前端工程：R8 必须拒绝生成“代码变更”，并提示用户提供正确路径。

### 1.3 输出（双产物：代码 + 对齐说明）

R8 的输出包含两部分：

1) **Demo 项目代码变更**（写入 DEMO 工程目录内）

- 新增/更新与线框原型对应的页面文件、路由入口、必要的占位内容与基础导航（用于走查）
- 所有生成内容必须带有显式标记（例如文件头注释 `@generated by aisdlc R8`），便于识别与回滚

2) **对齐说明落盘**（写入 Spec Pack）

- `{FEATURE_DIR}/implementation/demo.md`
- 该文件用于“证据入口 + 可追溯映射 + 走查清单”，其内容结构见第 5 节

### 1.4 职责边界（避免把 DEMO 当生产实现）

R8 必须做到：

- **可运行**：能启动、能访问到生成的页面入口
- **可走查**：页面结构与线框一致，关键控件/文案可见，关键跳转能跑通
- **可追溯**：能从 `prototype.md` 的页面编号（P-xxx）追溯到 DEMO 的路由/入口

R8 不负责：

- 对接真实后端接口、实现真实业务逻辑
- 做完整设计系统落地或视觉还原
- 解决工程架构选型（状态管理/请求层/权限系统等）

## 2. 强制门禁与总流程（关键约束：为避免“跳步/污染 DEMO 工程”而设）

### 2.1 强制门禁（MUST）

- **必须存在 `prototype.md`**。若缺失：
  - **不得**对 DEMO 工程做任何代码变更
  - 只能提示用户先完成 R7
- **必须拿到 DEMO 项目根路径**。获取优先级：先从 `project/memory/structure.md` 自动获取；若无法获取则要求用户提供。未提供或路径不可用：
  - **不得**写入任何 DEMO 文件
  - 只能提示用户补充路径
- **变更范围必须可隔离、可回滚**：
  - 生成文件应集中在一个可控的命名空间目录下（见 3.4）
  - 必须生成“变更清单”（manifest）并写入 `{FEATURE_DIR}/implementation/demo.md`
- **全程使用中文**；不确定信息必须标注为“未知/待确认”，不得脑补。

### 2.2 总流程（MUST）

1. 获取上下文：`Get-SpecContext` → 得到 `FEATURE_DIR`
2. 读取输入：`prototype.md` +（可选）`interaction.md/prd.md/structure.md/glossary.md` +（可选）上轮 `demo.md`
3. 获取 DEMO 根路径：
   - 优先从 `project/memory/structure.md` 自动获取/推断
   - 若无法获取：提示用户手动指定“前端 DEMO 项目根路径”，在用户提供前不得生成任何代码变更
4. 校验 DEMO 根路径：识别是否为前端工程、是否可写、是否可运行（至少能识别入口文件与路由方式）
4. 解析原型信息（最小集合）：
   - 页面/节点清单（至少 P-xxx）
   - 每页：页面标题、布局分区、关键控件/字段/操作、关键状态（正常/加载/空/错/无权限）
   - 页面跳转关系（若 `prototype.md` 不完整，可从 `interaction.md` 补齐；补不齐则标“待确认”并在 DEMO 用占位导航连接）
5. 选择生成策略（路由与目录）：
   - 自动探测 DEMO 工程的路由/页面目录（优先）
   - 若无法可靠探测：进入“最小补齐”问答（≤3 问，见第 4 节）
6. 生成/更新 DEMO 页面：
   - 创建页面文件（含占位内容、基础状态展示、主要操作按钮）
   - 建立入口与导航（例如一个 DEMO 目录页/侧边栏/路由表）
7. 落盘对齐说明：写入 `{FEATURE_DIR}/implementation/demo.md`
8. 自检：执行 R8-DoD（见第 6 节），并在 `demo.md` 标注未满足项与影响

## 3. 从线框到 DEMO 的生成规则（核心）

### 3.1 页面到路由的映射规则（建议固定）

R8 必须建立稳定映射：`prototype.md` 的页面编号（P-xxx） → DEMO 路由（URL/入口）。

建议默认规则（可按工程约束调整）：

- 路由前缀：`/aisdlc/{SPEC_NUMBER}/{page}` 或 `/spec/{CURRENT_BRANCH}/{page}`
- 页面 key：从页面名称生成 slug；若无法可靠生成，则使用 `p-001` 形式的稳定 key

约束：

- 同一需求分支（`CURRENT_BRANCH`）重复运行 R8 时，路由必须尽量稳定，避免走查链接频繁变化。

### 3.2 页面内容生成规则（占位但不空洞）

每个 DEMO 页面至少包含：

- 页面标题（与 `prototype.md` 一致）
- 布局分区标题（Header/Body/Footer/侧栏等）
- **信息与控件**（用占位实现“可见、可点击、可填写”）：
  - 展示字段：label + mock value（或 `—`）
  - 输入字段：可输入控件（文本框/下拉/日期等，无法判断类型时用文本框并标注“待确认”）
  - 操作按钮：主要按钮（提交/保存/删除/生成等），点击后给出“DEMO 提示反馈”
- **状态覆盖最小集**：
  - 正常态（默认）
  - 加载态（可用一个开关按钮切换）
  - 空态（可用一个开关按钮切换）
  - 错误态（可用一个开关按钮切换）
  - 无权限态（可用一个开关按钮切换）

> 原则：状态覆盖以“走查可见”为目标，不实现真实数据拉取。

### 3.3 跳转与链路闭环规则（可走通）

R8 需要为核心场景提供“可走通”的导航：

- 若 `prototype.md` 已定义跳转：按其建立页面间链接
- 若 `prototype.md` 未定义但 `interaction.md` 可推导：按交互链路补齐链接
- 若仍不确定：允许在 DEMO 中提供“占位跳转”，但必须：
  - 在 UI 上标注“待确认跳转”
  - 在 `{FEATURE_DIR}/implementation/demo.md` 的“未覆盖/待确认清单”中列出，并说明影响（影响场景/页面编号）

### 3.4 变更隔离与回滚策略（必须可控）

为避免污染 DEMO 工程与便于回滚，建议生成内容遵循：

- **集中目录**：所有新增页面放在固定目录下，例如：
  - React/Vite/CRA：`src/aisdlc/{CURRENT_BRANCH}/pages/*`
  - Next.js（App Router）：`app/aisdlc/{CURRENT_BRANCH}/*`
  - Next.js（Pages Router）：`pages/aisdlc/{CURRENT_BRANCH}/*`
- **单入口**：新增一个 DEMO 入口页（目录页/索引页），列出本需求所有页面链接与状态切换说明
- **生成标记**：所有生成文件在文件头注释标注：
  - 生成来源（R8）
  - 关联 `CURRENT_BRANCH`
  - 生成时间（可选）
  - 对应 `prototype.md` 页面编号（P-xxx）
- **Manifest 清单**：
  - 记录所有新增/修改的文件路径（用于回滚）
  - 记录页面映射（P-xxx → route → file path）
  - Manifest 内容必须落到 `{FEATURE_DIR}/implementation/demo.md`

### 3.5 增量更新原则（避免“每次全量重写”）

若存在上轮生成结果（通过文件头标记与 manifest 识别）：

- 默认 **增量更新**：仅对新增页面补齐、对已存在页面做最小改动（例如补充缺失分区/控件/跳转）
- 避免删除用户手工补充的内容；若确需覆盖，必须在 `demo.md` 记录“覆盖原因与范围”

## 4. 信息不足时的最小补齐（可选，但必须限量）

当无法自动探测 DEMO 工程的路由/页面结构时，允许最小补齐，但必须遵守：

- **最多 3 个问题**，且每次只问 1 个问题（与 R7 风格一致）
- 优先问题顺序：
  1) DEMO 工程使用的路由方式/框架（例如 Next.js App Router / Next.js Pages Router / React Router / Vue Router）
  2) 页面文件应生成到哪个目录（相对 DEMO 根目录）
  3) 路由入口文件/路由表位置（若需要修改）

用户未回答时：

- 必须停止代码生成（避免猜测写错工程结构）
- 只输出“待确认清单与建议”到 `{FEATURE_DIR}/implementation/demo.md`

---

## 5. `{FEATURE_DIR}/implementation/demo.md` 结构规范（建议保持稳定）

该文件是 R8 的“可追溯与走查入口”，建议稳定结构如下：

1) 概述（本次生成目标、DEMO 边界声明）  
2) 输入与证据入口（读取的 `prototype.md`、DEMO 根路径、可选引用的 `interaction/prd`）  
3) 生成策略摘要（探测到的框架/路由方式、生成目录、路由前缀规则）  
4) 页面映射表（必须）：
   - `P-xxx` → `页面名` → `DEMO 路由` → `文件路径` → `入口说明`
5) Manifest（文件变更清单，必须）：
   - 新增文件列表
   - 修改文件列表（若有）
6) 走查清单（建议）：
   - 核心场景链路走查步骤（按 S-xxx 或关键链路描述）
   - 每页的状态切换检查点（加载/空/错/无权限）
7) 未覆盖/待确认清单（必须有，哪怕为空）：
   - 待确认跳转
   - 待确认字段类型/规则/权限口径
   - 对走查与后续实现的影响说明
8) 回流建议（把问题落回 R6/R7/PRD）：
   - 该回流哪些文件、为什么、建议怎么改

## 6. R8-DoD（完成标准）

- [ ] 门禁通过：存在 `{FEATURE_DIR}/requirements/prototype.md`，且拿到 DEMO 根路径
- [ ] 输出落盘：生成/更新 `{FEATURE_DIR}/implementation/demo.md`
- [ ] 可追溯：`P-xxx → DEMO 路由/文件` 映射表完整
- [ ] 可回滚：manifest 中包含全部新增/修改文件路径
- [ ] 可走查：至少能访问“入口页/目录页”，并从中打开各页面
- [ ] 链路闭环：核心场景至少 1 条链路能在 DEMO 中走通（跳转缺失必须列入“待确认清单”）
- [ ] 状态可见：每个页面至少具备正常/加载/空/错/无权限的可见切换方式
- [ ] 明确边界：DEMO 属性与非生产边界在入口页与 `demo.md` 中显式声明

## 7. 提示词资产分层（用于约束未来命令文档结构）

建议将未来 `spec-product-demo` 的提示词按层组织（与 R7 风格对齐）：

- **提示词 0（入口门禁）**：校验 `prototype.md` 与 DEMO 根路径；不满足则拒绝写入 DEMO
- **提示词 1（解析与映射）**：从 `prototype.md` 抽取页面清单、跳转关系、状态覆盖，并生成 `P-xxx → route` 映射草案
- **提示词 2（工程探测）**：识别路由框架/目录结构，选择生成策略；必要时触发“最小补齐”
- **提示词 3（页面生成）**：逐页面生成占位内容、状态切换与跳转链接，并写入命名空间目录
- **提示词 4（对齐说明与校验）**：生成 `demo.md`（映射表 + manifest + 走查清单 + 待确认清单），并执行 R8-DoD 自检

> 设计约束：提示词内容可演进，但其职责边界、门禁规则、输出形态不得破坏本规范。

## 8. 下一步与闭环回流（执行完 R8 后）

R8 完成后建议立即进行一次 DEMO 走查，并按影响回流：

- **跳转链路不闭环 / 节点缺失 / 状态覆盖不足**  
  - 回流更新：`{FEATURE_DIR}/requirements/prototype.md`（R7）与必要时 `interaction.md`（R6）  
  - 重新运行：`spec-product-prototype` → 再运行 `spec-product-demo`
- **AC 不可验收 / 规则不一致 / 范围或里程碑冲突**  
  - 回流更新：`{FEATURE_DIR}/requirements/prd.md`（R5）  
  - 重新运行：`spec-product-prd` → `spec-product-interaction` → `spec-product-prototype` → `spec-product-demo`
- **场景/边界口径变化**  
  - 回流更新：`{FEATURE_DIR}/requirements/clarify.md`（R1）与必要时 `solutions.md`（R3）  
  - 重新运行：按主链路重跑下游模块，保持一致
