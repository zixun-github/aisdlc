## 0. 目标与定位

本设计文档用于细化 Spec 级需求分析中的 R6「场景交互方案」：在 R5 产出 `prd.md` 后，将 PRD 中的**核心场景/用户故事/业务规则/异常边界/验收标准（AC）**转写为**可评审、可落地、可进入原型（R7）的交互方案**，并落盘到：

- `{FEATURE_DIR}/requirements/interaction.md`

核心目标：

- **任务流可落地**：每个核心场景都有端到端用户任务流（含关键分支/回退/中断）
- **页面结构清晰**：从入口到完成，页面/弹窗/组件节点清单明确且可追溯
- **状态覆盖完整**：至少覆盖空/加载/错误/权限不足/无数据等关键状态与反馈
- **与 AC 一一对应**：交互节点支持验收（可指出“在哪个页面/哪个节点/哪个状态验证哪条 AC”）

> 约束：本模块不产出“视觉稿”，也不输出完整原型线框（下沉到 R7）；但必须输出足够细的交互与状态，确保研发/测试能准确理解实现与验收口径。

## 1. 术语与接口（命令必须遵守）

### 1.1 上下文与路径

- **FEATURE_DIR**：由 `Get-SpecContext` 自动获取，定位到 `.aisdlc/specs/{num}-{short-name}/`
- **必读输入**：
  - `{FEATURE_DIR}/requirements/prd.md`
- **可选输入**（用于补齐“情景摘要/痛点快点口径/术语一致性”）：
  - `{FEATURE_DIR}/requirements/clarify.md`
  - `{FEATURE_DIR}/requirements/solutions.md`
  - `.aisdlc/project/memory/glossary.md`
  - 既有 `{FEATURE_DIR}/requirements/interaction.md`（若存在：用于增量更新/对照）
- **输出**：`{FEATURE_DIR}/requirements/interaction.md`

### 1.2 输出产物的职责边界

`interaction.md` 的职责是把 PRD 的“场景”落到**交互可执行规格**，必须回答：

- **场景清单**：与 PRD 对齐（可编号）
- **情景摘要**：每个场景的标题/描述/痛点与快点（优先引用 `clarify.md`，若缺失可从 PRD 归纳并标注来源）
- **端到端任务流**：入口 → 关键步骤 → 完成条件 → 关键分支/异常/回退
- **页面/节点清单**：页面、弹窗、抽屉、二次确认、提示条、空态等（带 Node ID）
- **关键交互细节**：校验规则、提示文案、反馈策略、权限/风控/二次确认、失败处理
- **AC 映射**：交互节点如何支持 PRD 的验收（可输出映射表）

不属于 `interaction.md` 的内容：

- 视觉设计规范、颜色/字号/组件样式（交给 UI 规范）
- 可点击原型或线框图（交给 R7 的 `prototype.md` / Figma）
- 重新发散方案/新增需求（除“最小补齐”外不得引入新功能）

### 1.3 关键定义（建议固定口径）

- **场景（Scenario）**：围绕一个用户目标的端到端任务（可直接对应 PRD 的“核心场景/用户故事”组织方式）
- **任务流（Task Flow）**：用户为完成目标需要经历的步骤与分支（建议 Mermaid 流程图）
- **节点（Node）**：任务流中的“交互承载点”，通常是页面/弹窗/抽屉/二次确认/提示反馈等
- **状态（State）**：某个节点在不同数据/网络/权限/异常条件下的表现（如：加载、空态、错误、无权限）
- **交互节点（Interaction Point）**：对实现与验收有决定性影响的操作点（如：提交、撤销、删除、审批、导出）

## 2. 强制门禁与总流程（关键约束：为避免“跳步/脑补”而设）

### 2.1 强制门禁（MUST）

- **必须存在 `prd.md`**。若缺失：
  - **不得**输出 `{FEATURE_DIR}/requirements/interaction.md` 正文（即使是草稿）
  - 只能提示用户先完成 R5
- **全程使用中文**；不确定信息必须标注为“未知/待确认”，不得脑补。
- **不得新增需求**：任何 PRD 未明确且无法从 `clarify/solutions` 追溯的新增能力，必须标记为“待确认”并说明影响（影响哪条 AC/哪段任务流）。

### 2.2 总流程（MUST）

1. 获取上下文：`Get-SpecContext` → 得到 `FEATURE_DIR`
2. 读取输入：`prd.md` +（可选）`clarify.md/solutions.md/glossary.md` +（可选）上轮 `interaction.md`
3. 抽取场景：基于 PRD 的“核心场景/用户故事/AC”生成场景清单，并建立场景编号
4. 为每个场景生成交互：情景摘要 → 任务流 → 页面/节点清单 → 状态/异常 → 校验与文案要点
5. 建立追溯：输出“AC → 交互节点”的映射表；必要时补充“规则/异常 → 节点状态”的引用
6. 落盘：输出 `requirements/interaction.md`（只输出 Markdown 正文）
7. 自检校验：执行 R6-DoD（见第 6 节）

## 3. 从 PRD 到交互的结构化转写规则

### 3.1 场景抽取与对齐（Scenario List）

抽取原则：

- **以 PRD 为主**：以 PRD 的“核心场景与用户故事”“验收标准（AC）”作为主索引
- **可验收优先**：优先输出拥有 AC 的场景；若 PRD 的 AC 跨多个场景，应拆分为多个可交付场景或在场景内明确子流程
- **按用户目标分组**：避免把“功能点列表”直接当场景；场景应是用户能理解的任务（如“提交申请”“查询记录”“审批通过/驳回”）

建议输出场景编号：

- `S-001`、`S-002`…（稳定编号便于追溯与后续变更）

### 3.2 情景摘要（绑定痛点/快点，避免“只画流程”）

每个场景建议至少包含：

- **情景标题**：一句话说明任务（用户视角）
- **情景描述**：触发条件 + 目标 + 成功标准（可引用 PRD 用户故事）
- **痛点与快点**：
  - 痛点：现状问题（效率/出错/不可追溯/审批慢等）
  - 快点：希望的体验（一步完成/可追溯/明确反馈/减少反复等）

来源策略：

- 若存在 `clarify.md`：优先引用其“情景五要素”的表达作为来源（保持口径一致）
- 若仅有 PRD：允许从 PRD 摘要归纳，但必须在小节末标注“来源：PRD 第 X 节/AC-xxx”

### 3.3 端到端任务流（Task Flow）

必须覆盖：

- **入口**：用户从哪里进入（导航入口/列表入口/通知入口/链接入口）
- **关键步骤**：用户需要做哪些关键操作（输入/选择/提交/确认/查看结果）
- **关键分支**：
  - 权限分支（无权限、只读、可编辑）
  - 数据分支（无数据、部分缺失、数据过期）
  - 流程分支（成功/失败、需二次确认、需补充材料）
- **回退与中断**：取消/返回/保存草稿/关闭弹窗等是否允许与规则
- **完成条件**：成功完成后用户看到什么、能做什么下一步

建议：

- Mermaid 流程图优先（`flowchart TD`），每个场景至少 1 张图
- 图中节点 ID 与“页面/节点清单”保持一致（例如 `P-010`、`D-003`）

### 3.4 页面/弹窗/组件节点清单（Node Inventory）

节点类型建议统一编号：

- **页面（Page）**：`P-xxx`
- **弹窗/对话框（Dialog/Modal）**：`D-xxx`
- **抽屉（Drawer）**：`W-xxx`
- **提示/反馈（Toast/Banner/Inline）**：`T-xxx`

每个节点至少描述：

- **目的**：该节点解决什么用户意图
- **入口**：从哪个节点/哪个操作进入
- **关键区块**：信息展示区、输入区、操作区（无需做视觉布局，但要写清内容与控件）
- **关键控件与规则**：字段校验、默认值、禁用条件、提交条件
- **跳转**：成功/失败/取消/关闭/返回分别去哪

### 3.5 状态与反馈覆盖（State & Feedback）

每个“关键页面节点”至少覆盖以下状态（可按需裁剪，但需说明原因）：

- **正常态**：数据完整可操作
- **加载态**：首屏加载、列表分页加载、提交中（含按钮禁用/防重复）
- **空态**：无数据/无搜索结果（含引导动作与文案）
- **错误态**：
  - 网络错误（可重试）
  - 服务端错误（提示与工单/反馈入口，如适用）
  - 业务校验失败（字段级/表单级/全局）
- **权限不足**：不可见/可见但不可操作/只读（PRD 若未定义需标为待确认）

高风险操作（如删除、提交后不可逆、审批通过不可回退）必须：

- 有**风险提示**与**二次确认**策略（若 PRD 已规定则严格遵循；否则标为待确认并说明影响）

### 3.6 校验规则与提示文案（Validation & Copy）

交互方案必须把 PRD 的业务规则转成“可实现 + 可测试”的校验表达：

- **条件**（何时触发）+ **规则**（限制/范围/格式）+ **反馈**（提示位置与文案建议）

文案要求：

- 尽量“可行动”（告诉用户该怎么做），避免只给错误码或笼统报错
- 涉及术语必须与 `glossary.md` 一致（若不一致需标注并提出统一建议）

### 3.7 与验收标准（AC）的映射方法（Traceability）

建议在 `interaction.md` 末尾输出两张表：

1) **AC → 交互节点映射表**（必须）

- `AC-ID/描述` → `场景 S-xxx` → `节点 P/D/W-xxx` → `验证点（状态/文案/按钮可用性/跳转结果）`

2) **关键规则/异常 → 节点状态映射表**（可选但推荐）

- `规则/异常` → `触发条件` → `受影响节点` → `期望反馈`

## 4. 信息不足时的最小补齐（可选）

当 `prd.md` 无法支持“交互可落地/AC 可追溯”时，允许最小补齐，但必须遵守：

- **只补齐交互落地的必要信息**，不引入新需求
- **最多 3 个问题**，优先问：
  - 入口与用户角色/权限口径
  - 关键不可逆操作是否需要二次确认与提示口径
  - 关键异常处理口径（失败后是否可重试/是否落草稿/是否需要人工介入）
- 用户未回答时：
  - 仍可输出交互方案，但必须在对应节点标记“未知/待确认”并说明影响（影响哪条 AC/哪个环节）

## 5. `interaction.md` 结构规范（建议保持稳定，便于评审与进入原型）

建议以以下结构输出（与 `design/aisdlc_spec_product.md` 的 R6 口径一致）：

1) 概述（范围、输入来源、术语引用）  
2) 场景清单（与 PRD 对齐，S-001…）  
3) 场景详情（每个场景一节，建议固定模板）  
   - 情景摘要（标题/描述/痛点/快点/来源）  
   - 端到端任务流（Mermaid）  
   - 页面/节点清单（P/D/W/T 编号）  
   - 关键交互细节（校验/反馈/异常/权限/二次确认）  
4) AC 映射表（AC → 场景 → 节点 → 验证点）  
5) （可选）关键规则/异常 → 节点状态映射  
6) 追溯链接（指向 PRD/clarify/solutions/prototype）

场景详情模板（建议固定字段，便于审阅）：

- 场景编号：S-xxx  
- 触发与入口：  
- 前置条件：  
- 主流程：  
- 分支与异常：  
- 完成与后续动作：  
- 涉及节点：P/D/W/T-xxx  
- 关键规则与校验：  
- 状态覆盖：正常/加载/空/错/无权限  
- 对应 AC：AC-xxx…

## 6. R6-DoD（完成标准）

- [ ] `interaction.md` 的场景清单与 PRD 的核心场景/AC 一一对应（可追溯）
- [ ] 每个核心场景至少有 1 张端到端任务流（含关键分支/异常/回退）
- [ ] 页面/节点清单完整，且任务流节点 ID 与清单一致
- [ ] 异常与状态覆盖完整（至少包含空/加载/错误/权限不足；高风险操作含二次确认/提示策略）
- [ ] 每个核心场景至少标注 1 个痛点与 1 个快点，并能在交互节点中看到对应的设计策略
- [ ] 存在 AC → 交互节点映射表，能明确“在哪验证哪条 AC”

---

## 7. 提示词资产分层（用于约束未来命令文档结构）

为确保“可落地交互”但不“跳步/脑补”，建议提示词按层次组织：

- **提示词 0（入口门禁）**：检查 `prd.md` 是否存在；不满足门禁则拒绝生成交互正文
- **提示词 1（场景抽取与编号）**：从 PRD 抽取场景清单与 AC，并建立追溯索引
- **提示词 2（逐场景生成交互）**：按模板生成情景摘要/任务流/节点清单/状态与异常/校验与文案
- **提示词 3（追溯与校验）**：生成 AC 映射表 + 执行 R6-DoD 自检并标注缺口

> 设计约束：提示词内容可演进，但其职责边界、门禁规则、输出形态不得破坏本规范。

## 8. 下一步命令指引与闭环（执行完 `spec-product-interaction` 后）

当 R6 命令 `spec-product-interaction` 完成、并已落盘 `{FEATURE_DIR}/requirements/interaction.md` 后，下一步进入原型阶段：

- **下一步命令（R7）**：执行 `spec-product-prototype`  
  - **产物**：`{FEATURE_DIR}/requirements/prototype.md`（节点编号建议复用本模块的 `P/D/W/T-xxx`）

### 8.1 闭环回流（来自原型评审/可用性验证）

若在 R7 原型评审/最小可用性验证中发现关键问题，建议按影响优先级回流，并在回流后**重新运行对应命令**以保持下游一致：

- **场景口径 / 边界 In/Out / 痛点快点变化** → 回流更新 R1（`clarify.md`）→ 重新运行 `spec-product-clarify`（必要时再跑 R3/R5/R6）
- **AC 不可验收 / 规则不一致 / 范围漂移** → 回流更新 R5（`prd.md`）→ 重新运行 `spec-product-prd` → 再运行 `spec-product-interaction`
- **任务流 / 节点清单 / 状态覆盖不合理** → 回流更新本模块产物（`interaction.md`）→ 重新运行 `spec-product-interaction`
