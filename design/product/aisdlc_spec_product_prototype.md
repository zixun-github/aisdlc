## 0. 目标与定位

本设计文档用于细化 Spec 级需求分析中的 R7「原型生成」：在 R6 产出 `interaction.md` 后，将交互方案转写为**可评审、可被研发/测试准确实现与验收**的“原型资产”（文本原型/线框原型说明），并落盘到：

- `{FEATURE_DIR}/requirements/prototype.md`

核心目标：

- **页面结构可理解**：页面/弹窗/抽屉等节点清单清晰、编号稳定、可追溯到 R6 的任务流节点
- **状态与规则可实现**：正常/加载/空/错/无权限等关键状态明确；校验规则与提示文案可实现、可测试
- **验收可追溯**：可指出“哪个页面/哪个状态/哪个交互点支持哪条 AC”
- **可闭环**：在原型阶段输出最小可用性验证计划（任务脚本 + 回流指引），避免原型变成一次性产物

> 约束：本模块不产出视觉稿（颜色/字号/组件样式），也不强制依赖 Figma；若团队使用 Figma，`prototype.md` 仍必须保留“页面清单 + 状态说明 + 关键规则 + AC 追溯”，并提供 Figma 链接入口，避免口径漂移。

## 1. 术语与接口（命令必须遵守）

### 1.1 上下文与路径

- **FEATURE_DIR**：由 `Get-SpecContext` 自动获取，定位到 `.aisdlc/specs/{num}-{short-name}/`
- **必读输入**：
  - `{FEATURE_DIR}/requirements/interaction.md`
- **可选输入**（用于补齐 AC 追溯/口径一致性/场景脚本）：
  - `{FEATURE_DIR}/requirements/prd.md`
  - `{FEATURE_DIR}/requirements/clarify.md`
  - `{FEATURE_DIR}/requirements/solutions.md`
  - `.aisdlc/project/memory/glossary.md`
  - 既有 `{FEATURE_DIR}/requirements/prototype.md`（若存在：用于增量更新/对照）
- **输出**：`{FEATURE_DIR}/requirements/prototype.md`

### 1.2 输出产物的职责边界

`prototype.md` 的职责是把交互方案落成“可实现/可验收的原型说明”，必须回答：

- **页面编号与清单**：P-001…（建议稳定编号，便于评审与变更）
- **每个页面/节点**：
  - 页面目的与入口
  - 布局分区（Header/Body/Footer/侧栏等）
  - 信息与控件（字段、按钮、默认值、禁用条件）
  - 状态机（正常/加载/空/错误/权限不足）
  - 跳转与交互（点击/提交/返回/取消/二次确认）
  -（可选）埋点建议（与成功指标对齐，若需要）
- **关键页面线框**：用 ASCII / 表格 / 分区清单描述（不要求视觉，但要求结构与信息清晰）
- **追溯与验证**：
  - AC → 页面/状态/验证点映射（建议表格）
  - 最小可用性验证（任务脚本 + 记录方式 + 回流指引）

不属于 `prototype.md` 的内容：

- 视觉规范与 UI 组件样式（交给 UI 规范或设计系统）
- 新增未在 PRD/R6 体现的需求（除“最小补齐”外不得引入新功能）
- 把原型当 PRD 重写（业务背景/范围/里程碑下沉到 PRD）

### 1.3 编号与节点口径（建议固定）

建议复用 R6 的编号习惯，并在原型阶段稳定下来：

- **场景**：`S-001`、`S-002`…
- **页面（Page）**：`P-001`、`P-002`…
- **弹窗/对话框（Dialog/Modal）**：`D-001`…
- **抽屉（Drawer）**：`W-001`…
- **提示/反馈（Toast/Banner/Inline）**：`T-001`…
- **交互点（Interaction Point）**：`IP-001`…（可选；当需要精确追溯“按钮/提交点/二次确认点”）

> 约束：若 `interaction.md` 中已存在节点 ID（P/D/W/T），R7 必须优先复用；若未编号，R7 负责为其补齐并在 `prototype.md` 中建立“节点清单”，同时建议回流更新 `interaction.md` 以保持一致。

## 2. 强制门禁与总流程（关键约束：为避免“跳步/脑补”而设）

### 2.1 强制门禁（MUST）

- **必须存在 `interaction.md`**。若缺失：
  - **不得**输出 `{FEATURE_DIR}/requirements/prototype.md` 正文（即使是草稿）
  - 只能提示用户先完成 R6
- **全程使用中文**；不确定信息必须标注为“未知/待确认”，不得脑补。
- **不得新增需求**：任何 R6/PRD 未明确且无法追溯的新增能力，必须标记为“待确认”，并说明影响（影响哪条任务流/哪条 AC）。

### 2.2 总流程（MUST）

1. 获取上下文：`Get-SpecContext` → 得到 `FEATURE_DIR`
2. 读取输入：`interaction.md` +（可选）`prd.md/clarify.md/solutions.md/glossary.md` +（可选）上轮 `prototype.md`
3. 建立原型骨架：
   - 抽取场景清单（S-xxx）
   - 抽取/补齐节点清单（P/D/W/T）并稳定编号
4. 逐页面生成原型说明：
   - 页面目的/入口/前置条件
   - 布局分区与信息控件清单（含规则/默认值/禁用条件）
   - 状态机（至少覆盖正常/加载/空/错/无权限）
   - 跳转与交互（含二次确认/风控提示，如适用）
   - 线框描述（ASCII/表格）
5. 建立追溯：
   - AC → 页面/状态/验证点映射表（PRD 缺失则标注“AC 来源不足”的影响）
6. 输出最小可用性验证：
   - 任务脚本（按场景）+ 记录模板 + 回流指引（更新 R1/R5/R6/R3 的建议）
7. 落盘：输出 `requirements/prototype.md`（只输出 Markdown 正文）
8. 自检校验：执行 R7-DoD（见第 6 节）

## 3. 从交互到原型的结构化转写规则

### 3.1 场景与页面映射原则（避免“画了页面但丢了任务”）

- **以场景为主线**：`prototype.md` 的组织既要支持“按页面查看”，也要支持“按场景走读”。建议两者都提供入口：
  - 第一入口：场景清单（S-xxx → 对应页面链路）
  - 第二入口：页面清单（P-xxx → 目的/入口/状态）
- **节点来源唯一**：页面/弹窗等节点必须可追溯到 `interaction.md` 的任务流与节点清单；不得凭空新增页面。
- **链路闭环**：每个核心场景至少有一条“入口 → 操作 → 反馈 → 完成/回退”的闭环链路描述，并可定位到具体页面与状态。

### 3.2 页面编号策略（稳定性优先）

推荐策略：

- 若 R6 已定义 `P-xxx/D-xxx/W-xxx/T-xxx`：R7 完全复用
- 若 R6 未编号：
  - R7 在 `prototype.md` 中新增编号并保持稳定
  - 编号后建议回流更新 `interaction.md`，使任务流图与节点清单一致

编号稳定性约束：

- **禁止重排导致编号漂移**：除非发生大范围 IA 重构；小改动应追加页面（新编号）或在页面内扩展说明

### 3.3 页面模板（每页必须填写的最小字段）

每个页面/节点建议使用固定模板，至少包含：

- **ID & 名称**：P-xxx（页面名）/D-xxx（弹窗名）…
- **页面目的**：该节点解决什么用户意图
- **入口**：从哪个页面/哪个操作进入（引用上游节点 ID）
- **前置条件**：权限/数据/状态要求（未知则标“待确认”）
- **布局分区**：Header / Body / Footer / Sidebar（按需）
- **信息与控件清单**（必须结构化）：
  - 展示信息：字段、口径、来源
  - 输入控件：类型、默认值、可编辑条件、校验规则
  - 操作控件：按钮/菜单项、可用条件、二次确认策略
- **状态机**（至少 5 类，按需扩展）：
  - 正常态
  - 加载态（首屏/分页/提交中）
  - 空态（无数据/无结果）
  - 错误态（网络/服务端/业务校验）
  - 无权限态（不可见/只读/禁止操作）
- **跳转与交互**：
  - 成功后去哪（并产生什么反馈）
  - 失败后如何处理（可重试/保留输入/落草稿）
  - 返回/取消/关闭行为
-（可选）**埋点建议**：仅当 PRD 有成功指标/可观测要求时补充

### 3.4 状态与反馈规范（把“异常覆盖”落到页面上）

状态覆盖必须可实现且可测试，建议统一表达格式：

- **触发条件**：何时进入该状态
- **界面表现**：核心区域显示什么、可操作性如何（按钮禁用/可点）
- **反馈文案建议**：尽量可行动
- **恢复路径**：重试/返回/联系客服/提交工单（如适用）

提交类交互必须包含：

- 提交中按钮禁用/防重复
- 成功反馈（提示 + 状态刷新 + 下一步）
- 失败反馈（字段级/表单级/全局）与恢复方式

高风险/不可逆操作必须包含：

- 风险提示与二次确认（D-xxx）
- 二次确认文案要点（动作后果、不可逆提示、替代方案/撤销口径）

### 3.5 校验规则与提示文案的表达方式

建议写成“可测试三元组”：

- **条件**（触发时机）+ **规则**（限制/范围/格式）+ **反馈**（提示位置与文案建议）

示例（表达风格示意，不强制字段）：

- 条件：点击“提交”
- 规则：必填字段为空
- 反馈：字段下方红字提示“请填写xxx”；页面顶部提示条“还有 2 项未填写”

### 3.6 与 AC 的追溯（强烈建议输出）

建议在 `prototype.md` 中输出“AC → 页面/状态/验证点”表：

- `AC-ID/描述` → `场景 S-xxx` → `页面/节点 P/D/W-xxx` → `验证点（状态/文案/按钮可用性/跳转结果）`

若 PRD 不存在或 AC 缺失：

- 仍可输出原型，但必须明确标注“AC 来源不足”，并建议回流补齐 PRD（否则测试无法落口径）

### 3.7 原型信息表达协议（必填字段集）

本节用于解决一个常见问题：原型“看起来像线框”，但缺少**可实现/可验收**的信息，且难以用 AI 做**精准修改**。因此要求每个页面/节点必须同时表达“三件套”；当需要精确改动时，可补充“元素清单”（可选）。

**三件套（每个页面/节点必须同时具备）**：

1. **页面清单（Node Inventory）**：P/D/W/T 的稳定编号、用途、入口、涉及场景（可追溯到 `interaction.md`）。
2. **页面核心功能（Core Capabilities）**：每页 **3～7 条**，必须“动词开头 + 作用对象 + 结果/反馈”，尽量可验收。
3. **页面布局线框（Layout Wireframe）**：必须有**直观的 ASCII 线框**（见 3.8），能一眼看懂布局、字段与操作；若需要 AI 做精准调整，可在页面详情里补充“元素清单”（可选）。

**核心功能（Core Capabilities）写法规范**（建议固定句式）：

- 句式：`<动词> <对象>，<结果/反馈>（可选：约束/规则）（可选：对应 AC-xxx）`
- 示例：
  - `搜索 订单列表，返回匹配结果并高亮关键词（AC-003）`
  - `提交 表单，成功后返回列表并提示“已提交”（AC-007）`
  - `删除 条目，需要二次确认并说明撤销/恢复口径（待确认：是否可撤销）`

**线框表达最低要求**：

- 线框必须包含：页面标题、关键分区、关键字段、关键操作（尤其是提交/删除/生成等）。
- 线框必须体现：关键提示/校验信息大致出现的位置（例如字段下方、页面顶部提示条、弹窗等）。
- **直观优先**：线框图中不强制写一堆 ID（避免影响可读性）；如需 AI 精准调整，使用 3.8.4 的“元素清单”（可选）。

### 3.8 ASCII 线框规范（直观优先）

本节只保留“原则与最小要求”，避免把设计文档写成一份冗长的线框图库；**详细的线框书写约定与可复制的页面类型示例库**已迁移到命令提示词中，以确保生成时可直接遵守与复用：

- `.aisdlc-cli/commands/spec-product-prototype.md` → `提示词 2. 逐页面生成原型`（包含：线框书写约定、列表/详情/表单/向导示例库、mock 数据建议）

ASCII 线框的**最小要求（MUST）**：

- **必须包含**：页面标题、关键分区标题、关键字段/信息、关键操作（尤其提交/删除/生成等）、关键提示/校验信息的大致位置。
- **必须一致**：仅使用纯 ASCII（`+ - |` 字符画），不使用表格线框，保证在任何环境下呈现一致。
- **直观优先**：线框中优先使用“人类可读名称”（分区/字段/按钮名），不强制把 ID 挤进图里。
- **可精确修改（可选）**：如需要 AI 精准调整，建议在页面详情补充“元素清单（ElementKey）”用于点名定位（线框图保持直观即可）。

### 3.9 页面类型线框示例库（可复制粘贴）

本节中的“示例线框库”已迁移到命令提示词中，避免设计文档承载过多可复制的具体线框细节，同时确保生成时可直接复用：

- `.aisdlc-cli/commands/spec-product-prototype.md` → `提示词 2. 逐页面生成原型`（常见页面类型线框示例库）

设计文档层面的约束只保留两点：

- **示例不构成新增需求**：实际页面必须以 `interaction.md` 的节点为准，禁止为了“套示例”而新增页面。
- **表达协议一致**：线框必须满足 3.8 的最小要求（标题/分区/字段/操作/提示位置），并与模板结构保持一致。

### 3.10 AI 调整指令规范（Diff-friendly / 可点名修改）

为了让后续“用 AI 调整原型”可控、可追溯，本节定义一种**点名到页面-区域-控件/字段/动作**的变更写法。该写法不是必须落盘到 `prototype.md`，但建议在评审讨论/需求变更时使用。

#### 3.10.1 变更指令格式（建议）

每条变更一行，推荐两种写法（二选一）：

- **写法 A（直观路径）**：`Change: <NodeID> > <分区标题> > <控件/字段/操作名称> <Operation> ; <Key>=<Value>`
- **写法 B（元素清单键）**：`Change: <NodeID>@<ElementKey> <Operation> ; <Key>=<Value>`

其中 `<Operation>` 建议枚举：`add` / `remove` / `move` / `rename` / `update_rule` / `add_state` / `update_state` / `update_copy`

示例（表达意图，不代表新增需求）：

- `Change: P-001 > 权限范围 > 订单管理 > 导出订单数据 update_copy ; copy="导出订单（CSV）"`
- `Change: P-001@E_Actions.Generate update_rule ; disable_when=Loading ; anti_repeat=true`
- `Change: P-List > 筛选 > 时间 add ; type=dropdown ; options=近7天,近30天,自定义`
- `Change: P-List add_state ; state=Empty ; copy="暂无数据，去新建订单" ; show=新建订单按钮`

#### 3.10.2 变更约束（必须遵守）

- 不得引入 `interaction.md/PRD` 未定义且不可追溯的新需求；若确需提出，必须写为 `待确认` 并说明影响链路（影响场景/页面/AC）。
- **先改“可定位信息”再改线框**：优先补齐/更新“元素清单（可选）”或明确到“分区标题/控件名/字段名/操作名”，再调整线框与文案，避免“改了线框但无法精确追溯”。
- 任何变更都应可追溯到：`S-xxx`、`P/D/W/T-xxx`、（如有）`AC-xxx`。

## 4. 信息不足时的最小补齐（可选）

当 `interaction.md` 无法支持“页面可实现/状态可测试”时，允许最小补齐，但必须遵守：

- **只补齐原型落地的必要信息**，不引入新需求
- **最多 3 个问题**，优先问：
  - 关键权限口径（无权限是不可见还是只读）
  - 高风险操作是否需要二次确认与文案口径
  - 提交失败后的恢复口径（是否保留输入/是否落草稿）
- 用户未回答时：必须在对应页面/状态标记“未知/待确认”，并说明影响（影响实现/验收/AC 追溯）

## 5. `prototype.md` 结构规范（建议保持稳定，便于评审与进入研发/测试）

建议以以下结构输出：

1) 概述（范围、输入来源、术语引用、Figma 链接入口如适用）  
2) 场景清单（S-001…，每个场景给出页面链路摘要）  
3) 页面/节点清单（P/D/W/T 编号、名称、用途、入口）  
4) 页面详情（逐页，按固定模板输出）  
5) AC 追溯表（AC → 场景 → 页面/状态 → 验证点）  
6) 最小可用性验证（任务脚本 + 记录模板 + 回流指引）  
7) 追溯链接（PRD/clarify/solutions/interaction）

线框表达建议（强制 + 可选）：

- **强制：纯 ASCII 线框（`+ - |` 字符画）**：表达分区与信息层级（与 `.aisdlc-cli/templates/spec/product-prototype-template.md` 对齐）
-（可选）**分区清单**：用“基本信息/有效期设置/权限范围/底部操作”等小标题列出字段与操作（当 ASCII 线框不够清晰时补充）
-（可选）**元素清单**：用于 AI 精准调整（线框图保持直观即可；元素清单写法建议见 3.8 的说明与命令提示词中的示例/约定）

## 6. R7-DoD（完成标准）

## 7. 提示词资产分层（用于约束未来命令文档结构）

为确保“原型可落地”但不“跳步/脑补”，建议提示词按层次组织：

- **提示词 0（入口门禁）**：检查 `interaction.md` 是否存在；不满足门禁则拒绝生成原型正文
- **提示词 1（抽取与编号）**：从 `interaction.md` 抽取场景/节点，并稳定编号（P/D/W/T）
- **提示词 2（逐页面生成原型）**：按页面模板输出布局分区、控件、规则、状态机、跳转与线框描述
- **提示词 3（追溯与校验）**：生成 AC 追溯表 + 执行 R7-DoD 自检并标注缺口
- **提示词 4（可用性验证）**：从核心场景生成任务脚本、记录模板与回流建议

> 设计约束：提示词内容可演进，但其职责边界、门禁规则、输出形态不得破坏本规范。

## 8. 下一步与闭环回流（执行完 `spec-product-prototype` 后）

当 R7 命令 `spec-product-prototype` 完成、并已落盘 `{FEATURE_DIR}/requirements/prototype.md` 后，建议立即做一次最小评审闭环：

### 8.1 下一步：原型评审 / 最小可用性验证

- 组织原型评审：研发/测试至少走读 1 遍核心场景（按 `prototype.md` 的场景脚本）
- 做最小可用性验证：按“任务脚本 → 记录问题 → 归因 → 回流修正文档”执行

### 8.2 回流路径（发现问题后怎么做才可执行）

在原型评审与验证中，若发现关键问题，建议按影响回流更新，并在回流后**重新运行对应命令**，保持下游一致（避免只改某一个文件导致口径漂移）：

- **任务流不合理 / 状态覆盖不足 / 节点缺失或冗余**  
  - 回流更新：`{FEATURE_DIR}/requirements/interaction.md`（R6）  
  - 重新运行：`spec-product-interaction` → 再运行 `spec-product-prototype`
- **AC 不可验收 / 规则不一致 / 范围或里程碑冲突**  
  - 回流更新：`{FEATURE_DIR}/requirements/prd.md`（R5）  
  - 重新运行：`spec-product-prd` → 再运行 `spec-product-interaction` → 再运行 `spec-product-prototype`
- **场景/边界口径变化（In/Out、痛点/快点、关键约束）**  
  - 回流更新：`{FEATURE_DIR}/requirements/clarify.md`（R1）  
  - 重新运行：`spec-product-clarify`（必要时再跑 `spec-product-solutions`）→ `spec-product-prd` → `spec-product-interaction` → `spec-product-prototype`
- **方案机制需要调整（关键规则/权衡/风险与验证）**  
  - 回流更新：`{FEATURE_DIR}/requirements/solutions.md`（R3）  
  - 重新运行：`spec-product-solutions` → `spec-product-prd` → `spec-product-interaction` → `spec-product-prototype`

### 8.3 回流原则（保持文档一致）

- **先修口径再修原型**：优先修 PRD/交互/方案的“口径与约束”，再更新原型，避免“原型领先但文档滞后”
