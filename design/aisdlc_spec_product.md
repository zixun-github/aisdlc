#  产品设计流程

## 1. 背景与目标（对齐 `design/aisdlc.md`）

本方案用于 **Spec 级** 的"需求分析阶段"提效：在遵循"双层 SSOT + Spec as Code + 渐进式披露"的前提下，用 **分布式、可独立优化的 AI 对话流程**，把一个模糊需求稳定推进到以下产物：

- **PRD**：可评审、可验收、可拆解、可追溯
- **场景交互方案**：以场景/任务为主线的流程与页面结构
- **原型**：低保真到中保真（文本原型/线框原型/可点击原型说明），可进入 UI 设计与研发
- **前端 DEMO（可选）**：基于线框原型生成可运行的页面与占位内容，用于走查与快速对齐（非生产实现）

> 说明：这里的"需求分析阶段"面向落盘产物以 `requirements/` 为主（PRD/用例/验收口径），交互与原型可以放在 `requirements/` 或 `design/`，取决于团队分工（建议见 1.1）。

## 2. 产物落盘与"渐进式披露"读取顺序

### 2.1. 建议落盘结构（需求级 Spec Pack）

对齐 `design/aisdlc.md` 中的需求级目录（示例）：

```text
.aisdlc/specs/<DEMAND-ID>/
  index.md
  requirements/
    raw.md                    # 原始输入（工单/访谈/截图/会议纪要摘要）
    clarify.md                # 需求澄清：痛点-场景-边界-问题清单
    competitors.md            # 竞品与替代方案分析
    solutions.md              # 最终方案（结构化）+ 覆盖校验 + 风险/验证 + 迭代记录
    prd.md                    # PRD（可评审版本）
    interaction.md            # 场景交互方案（信息架构/流程/关键页面说明）
    prototype.md              # 原型说明（页面清单、线框、交互细节、状态/校验）
  design/
    solution.md               # 如涉及架构/契约/ADR，再下沉到 design/
  implementation/
    demo.md                   # （可选）基于线框原型生成的前端 DEMO 页面对齐说明（概要）
```


### 2.2. Agent 读取顺序（渐进式披露）

- **必读（项目级）**：先读项目级 `memory/`（业务、技术、结构、术语）与相关 contracts/adr 索引，避免"业务口径/约束/边界"写错。
- **按需（需求级）**：仅在明确处理某个 `<DEMAND-ID>` 时读取该需求的 `requirements/*`，并尽量"最小必要读取"。
- **回写（入库）**：每个模块独立产出一个文件（或一个章节），保持可替换与可优化。

### 2.3. 上下文自动识别机制

对齐 `design/aisdlc_spec_init.md` 中的"上下文自动识别机制"要求，所有 spec 级命令在执行前，都必须先通过标准化流程获取当前 spec 相关的上下文信息。

**要做什么**：
- 调用脚本函数 `Get-SpecContext`（位于 `.aisdlc-cli/scirpts/spec-common.ps1`）
- 该函数会：
  - 获取当前 Git 分支名称（`git branch --show-current`）
  - 基于分支名称 `{num}-{short-name}` 构建 Spec 目录路径 `.aisdlc/specs/{num}-{short-name}/`
  - 验证分支格式和目录结构

**输出什么**：
- **REPO_ROOT**：Git 仓库根目录路径
- **CURRENT_BRANCH**：当前分支名称（格式：`{num}-{short-name}`）
- **FEATURE_DIR**：当前需求的 Spec Pack 根目录路径（`.aisdlc/specs/{CURRENT_BRANCH}/`）
- **SPEC_NUMBER**：分支编号部分（从分支名称提取）
- **SHORT_NAME**：分支短名称部分（从分支名称提取）

所有模块（R1-R8）的输入文件路径基于 `FEATURE_DIR` 自动构建，例如：
- `requirements/raw.md` → `{FEATURE_DIR}/requirements/raw.md`
- `requirements/clarify.md` → `{FEATURE_DIR}/requirements/clarify.md`
- `requirements/prd.md` → `{FEATURE_DIR}/requirements/prd.md`

## 3. 分布式流程总览（模块化流水线）

目标：每一步都能单独执行、单独评审、单独替换；同时步骤之间通过明确的输入/输出对齐，避免重复劳动与漂移。

| 模块 ID | 模块名称 | 主要目标 | 关键输出 |
|---|---|---|---|
| R0 | Spec 初始化（创建工作分支与 Spec Pack） | 建立“需求上下文”（分支+目录）并落盘原始需求，为后续模块提供稳定的输入路径与自动识别能力 | Spec 分支 `{num}-{short-name}` + `.aisdlc/specs/{num}-{short-name}/` + `requirements/raw.md` |
| R1 | 需求澄清（痛点与场景） | 把"要做什么/为什么做/对谁有价值/边界是什么"说清楚 | `requirements/clarify.md` |
| R2 | 竞品/替代方案分析 | 通过拆解竞品沉淀**可复用的细化功能清单（功能池）** + 关键流程/策略/约束 | `requirements/competitors.md` |
| R3 | 最终方案完善与迭代（基于用户方案输入） | 基于用户方案描述，完成结构化表达 → 覆盖校验（对齐 clarify）→ 细化补齐 → 风险/验证 → 迭代记录 | `requirements/solutions.md`（最终方案+覆盖校验+风险+验证+迭代记录） |
| R4 | 预留（不在本方案范围） | - | - |
| R5 | 基于方案生成 PRD | 把决策转为可交付规格（含 AC） | `requirements/prd.md` |
| R6 | 场景交互方案 | 把 PRD 的场景转为用户任务流与页面结构 | `requirements/interaction.md` |
| R7 | 原型生成（线框） | 把交互方案转为可评审的**线框原型**与状态说明 | `requirements/prototype.md` |
| R8 | 前端 DEMO 页面生成（可选） | 基于线框原型在前端 DEMO 项目中生成可运行页面与占位内容，用于走查与快速对齐（非生产实现） | Demo 项目代码变更 + `implementation/demo.md` |

### 3.2 命令式工作流（每个节点一个命令 + 执行后给出下一步）

本流程的**操作粒度**是“一个节点 = 一个命令 = 一个落盘产物”。为了让不熟悉流程的同学也能快速上手，建议把命令当作“路标”使用：**每执行完一个命令，立即根据本节的引导执行下一条命令**。

**命令与模块对照**（命令文档是实现约束 SSOT；主文档是端到端流程 SSOT）：

| 模块 | 运行的命令（同名命令文件） | 产物（落盘） | 下一步（建议） |
|---|---|---|---|
| R0 | `spec-init`（`.aisdlc-cli/commands/spec-init.md`） | `requirements/raw.md` | → R1：`spec-product-clarify` |
| R1 | `spec-product-clarify`（`.aisdlc-cli/commands/spec-product-clarify.md`） | `requirements/clarify.md` | → R2（可选）：`spec-product-competitors`；或 → R3：`spec-product-solutions` |
| R2 | `spec-product-competitors`（`.aisdlc-cli/commands/spec-product-competitors.md`） | `requirements/competitors.md` | → R3：`spec-product-solutions` |
| R3 | `spec-product-solutions`（`.aisdlc-cli/commands/spec-product-solutions.md`） | `requirements/solutions.md` | → R5：`spec-product-prd` |
| R5 | `spec-product-prd`（`.aisdlc-cli/commands/spec-product-prd.md`） | `requirements/prd.md` | → R6：`spec-product-interaction` |
| R6 | `spec-product-interaction`（`.aisdlc-cli/commands/spec-product-interaction.md`） | `requirements/interaction.md` | → R7：`spec-product-prototype` |
| R7 | `spec-product-prototype`（`.aisdlc-cli/commands/spec-product-prototype.md`） | `requirements/prototype.md` | → R8（可选）：`spec-product-demo`；或 → 原型评审/最小可用性验证 → 回流更新 R1/R3/R5/R6 |
| R8 | `spec-product-demo`（待实现） | Demo 项目页面 + `implementation/demo.md` | → DEMO 走查/最小可用性验证 → 回流更新 R1/R3/R5/R6/R7 |

> 说明：命令的“规范定义”以 `.aisdlc-cli/commands/` 为准；若你在 Cursor 中以“自定义命令/Command”方式运行，仓库里通常也会有对应的镜像命令文件（例如放在 `.cursor/commands/`），以仓库实际情况为准。

**最短路径（建议新手先跑通）**：

`spec-init` → `spec-product-clarify` → `spec-product-solutions` → `spec-product-prd` → `spec-product-interaction` → `spec-product-prototype` →（可选）`spec-product-demo`

**增强路径（信息不足/高风险/需行业对照时启用）**：

在 R1 与 R3 之间插入 R2：`spec-product-competitors`（用于沉淀“功能池/机制池”，帮助 R3 更快补齐默认策略、风险与取舍）。

下面这些文档是对各模块“门禁、接口、输出结构、DoD”的**硬约束规范**（主要用于约束命令实现与模板演进）；本主文档则负责端到端总览与落盘约定。

- **R1 需求澄清**：`design/product/aisdlc_spec_product_clarify.md`
- **R2 竞品/替代方案分析**：`design/product/aisdlc_spec_product_competitors.md`
- **R3 最终方案完善与迭代（基于用户方案输入）**：`design/product/aisdlc_spec_product_solutions.md`
- **R5 PRD**：`design/product/aisdlc_spec_product_prd.md`
- **R6 场景交互方案**：`design/product/aisdlc_spec_product_interaction.md`
- **R7 原型**：`design/product/aisdlc_spec_product_prototype.md`
- **R8 前端 DEMO（可选）**：`design/product/aisdlc_spec_product_demo.md`（路由约定、目录结构、Mock 数据策略、生成规则与回滚策略等）

> **闭环提示（把"原始需求"拆解为可落盘资产）**：可以把原始需求理解为"**设计目的 + 四大分析维度（目标用户 / 干系人 / 竞品 / 情景）**"。其中：
> - **目标用户/干系人**：优先在 R1 落盘（含决策链：谁使用、谁付费、谁验收/审批）。
> - **竞品**：在 R2 按"同痛点同情景"做分类、收集、拆解，并把机制整合为候选策略/功能。
> - **情景**：在 R1 用"情景五要素（要素罗列/情景标题/情景描述/痛点与快点/功能启发）"把场景说透，并在 R6 映射到任务流、页面与状态。
> - **功能列表与优先级**：在 R5 基于“用户方案输入 + clarify 覆盖校验 +（可选）竞品机制 + 方案约束”形成 MoSCoW 分级与 MVP（并落到范围/里程碑/AC）。
> - **验证与迭代**：在 R7 输出可评审的线框原型；必要时在 R8 生成可运行 DEMO 页面用于走查验证，并把结果回流更新 R1/R3/R6/R7。

## 3.1 模块 R0：Spec 初始化（创建工作分支与 Spec Pack）

### 3.1.1 目标

把“原始需求输入”快速转为**可追溯、可自动识别上下文**的 Spec 工作空间：通过创建 `{num}-{short-name}` 分支与标准化 Spec Pack 目录，为后续 R1-R8 提供稳定的文件路径与一致的执行上下文。

### 3.1.2 输入与触发

- **输入**：原始需求文本或原始需求文件路径（优先文件路径，以降低中文参数传递的编码风险）。
- **触发时机**：当还没有对应的 spec 分支/目录，或需要开启一个全新的需求 Spec Pack 时执行。

### 3.1.3 输出（概要）

- **分支**：创建并切换到 `{num}-{short-name}`（编号自增，命名规范统一）。
- **目录**：创建 `.aisdlc/specs/{num}-{short-name}/` 及标准子目录（`requirements/`、`design/`、`implementation/`、`verification/`、`release/`）。
- **原始需求落盘**：写入 `requirements/raw.md`（作为后续所有分析与设计的“证据入口”）。

### 3.1.4 与“上下文自动识别”的关系

R0 的核心价值之一是把 **Git 分支**作为需求级上下文标识符：后续模块在执行前通过 `Get-SpecContext` 获取 `FEATURE_DIR`，从而无需额外参数即可定位当前需求的 Spec Pack（详见 `design/aisdlc_spec_init.md`）。

### 3.1.5 关键约束（概要）

- **非破坏性**：不覆盖已存在的分支或目录（除非明确确认）。
- **一致性**：分支名与 Spec 目录名一一对应，保证自动识别可靠。
- **编码统一**：中文内容相关文件以 UTF-8 with BOM 写入，减少跨环境乱码风险。

### 3.1.6 执行命令（R0）

- **命令**：`spec-init`（见 `.aisdlc-cli/commands/spec-init.md`）
- **用户输入**：建议优先传入“原始需求文件路径”（降低中文参数传递的编码风险）；也支持直接粘贴文本
- **完成后检查**：
  - 当前分支已切换为 `{num}-{short-name}`
  - `.aisdlc/specs/{num}-{short-name}/requirements/raw.md` 已生成（UTF-8 with BOM）

### 3.1.7 下一步命令（R0 → R1）

执行 R0 后，立即进入 R1 做需求澄清：

- **下一步**：`spec-product-clarify`（生成 `requirements/clarify.md`）

## 4. 模块 R1：需求澄清（痛点与场景）

### 4.1. 目标

把输入的"原始需求"加工为可执行的澄清结论：**明确要做什么、为什么做、对谁有价值、边界是什么**。

### 4.2. 输入

**上下文自动获取**：
- 通过上下文自动识别机制获取 `FEATURE_DIR`
- 自动定位 `{FEATURE_DIR}/requirements/raw.md`（尽量保留原始措辞、证据链接/截图）

**项目级资源**：
- 项目级 `memory/glossary.md`（术语与口径，如有）

### 4.3. 输出（落盘到 `requirements/clarify.md`）
核心内容（按需扩展）：

- **设计目的**：为什么要做？核心价值是什么？
- **目标用户**：谁遇到问题？谁使用？谁决策/验收？（简单需求可合并为"用户"）
- **关键场景**：用户在什么情况下遇到问题？期望如何解决？（用 1-2 句话描述即可）
- **痛点**：现状是什么？为什么痛？（频率/影响可简要说明）
- **目标与非目标**（In/Out）：这次做什么？明确不做什么？
- **约束**：合规/权限/性能/时效等硬约束（如有）
- **待确认问题**：需要进一步澄清的关键问题（影响方案选择或验收的）

> **扩展场景**：对于复杂需求，可在"关键场景"中补充"情景描述"（故事化）、"痛点与快点"、"功能启发"等细节；对于简单需求，保持上述核心项即可。

### 4.4. 交互式补充（可选，但推荐“可追溯”）

R1 是需求分析中的关键步骤，常见问题是：信息不足、结论只存在于对话、追问无止境。建议采用“**可选交互式补充 + 增量写回 raw.md + 最后验证**”来保证质量与可追溯性。

**是否启用交互式补充**：

- 对于简单需求，用户可选择“否”跳过交互式补充，直接基于 `raw.md` 生成 `clarify.md`（缺信息写到“待确认问题”）。
- 对于复杂/高风险需求（合规/安全/审计/跨团队验收/范围易漂移），建议用户选择“是”执行交互式补充。

**交互式补充的标准流程**（仅用户选择“是”时执行）：

1. **生成候选澄清问题队列（≤5）**：结合 `requirements/raw.md` 与模板要求（`.aisdlc-cli/templates/spec/product-clarify-template.md`）生成“高信息增益”问题，并按影响验收/合规/范围/成本排序。
2. **顺序提问循环（一次一个问题）**：
   - 每次只问 1 个问题
   - 在顶部显著位置给出你的“推荐选项”并说明 1～2 句理由
   - 用户回答后判断是否已澄清；如仍有歧义，用最小补问消歧或调整队列
   - 具备停止机制：DoD 达标/队列耗尽/轮次上限/用户停止/边际收益过低
3. **每次接受答案后的集成（增量更新）**：把“问题 + 推荐选项 + 用户回答 + 一句话结论”追加到 `raw.md` 的“澄清记录”中，形成可追溯记录。
4. **重新读取更新后的 raw.md**：按模板稳定输出 `requirements/clarify.md`。
5. **验证（最后一遍执行）**：做记录完整性、一致性、模板可填充性与 R1-DoD 自检。

> 详细方案见 `design/product/aisdlc_spec_product_clarify.md`（包含问题队列约束、停止机制、raw.md 追加格式、验证清单与后续命令指引）。

### 4.5. AI 对话提问脚本（核心 3 问）

1. **目标与价值**：请用一句话复述需求目标，并说明"为什么要做"（用户目标 + 业务目标）。
2. **用户与场景**：谁遇到问题？在什么情况下遇到？期望如何解决？
3. **边界与约束**：这次明确做什么？不做什么？有哪些硬约束（合规/权限/时效等）？

> **按需扩展**：对于复杂需求，可追加"干系人优先级"、"量化证据"、"验收标准"等问题；对于简单需求，上述 3 问足够。

### 4.6. 质量门槛（R1-DoD）

- 目标与价值清晰（能回答"为什么做"）
- 用户与场景可复述且不歧义
- In/Out 明确（避免范围漂移）
- 约束已列出（未知也显式写“未知”）
- 待确认问题已列出（如有）
- 有证据/引用入口（raw/glossary 等，如适用）

### 4.7 执行命令（R1）

- **命令**：`spec-product-clarify`（见 `.aisdlc-cli/commands/spec-product-clarify.md`）
- **门禁（必须先过）**：命令必须先询问用户是否启用“交互式补充”（A 否 / B 是），在用户回答前不得生成 `clarify.md`
- **交互式补充开启时的硬约束**（摘自 `design/product/aisdlc_spec_product_clarify.md`）：
  - 问题队列 **≤5**，且对用户“一次只问 1 题”
  - 每轮回答后必须把“问题 + 推荐选项 + 用户回答 + 一句话结论 + 遗留歧义”**增量写回 `raw.md` 的 `## 澄清记录`**
  - 具备停止机制（DoD 达标 / 队列耗尽 / 轮次上限 / 用户停止 / 边际收益过低）
- **完成后检查**：
  - `requirements/clarify.md` 已生成且满足 R1-DoD

## 5. 模块 R2：竞品/替代方案分析

### 5.1. 目标

对齐《网龙产品设计方法论》的“竞品分析（Competitive Analysis）”，在**相同痛点/相近场景**下完成：

- **竞品拆解（核心）**：用“菜单拆解/流程拆解/重要操作拆解”的碎片化方式，把竞品功能**一一拆出来并落盘**，形成**细化功能清单（功能池）**。
- **流程与机制提炼**：沉淀主流程、关键分支/异常处理、引导/风控/权限等关键策略与 trade-off。
- **可复用输入**：让 R3 能直接复用 R2 的“功能池 + 关键机制/约束”进行方案发散与筛选，而不是停留在表层对比。

### 5.2. 输入

**上下文自动获取**：
- 通过上下文自动识别机制获取 `FEATURE_DIR`
- 自动定位 `{FEATURE_DIR}/requirements/clarify.md`

**用户输入**：
- 竞品列表（若未知，先由 AI 生成候选清单，再由人筛选）
-（推荐）每个竞品的**入口信息/证据入口**：官网链接、产品说明书、应用商店链接、可访问的截图/录屏、公开文档等（至少能支撑一次“流程拆解”或“重要操作拆解”）

### 5.3. 输出（落盘到 `requirements/competitors.md`）

建议结构（强调“拆解→功能清单→整合”为核心）：

- **对比基准**：用 R1 的场景作为基准（任务目标/触发/关键约束/成功标准），避免泛泛比较
- **竞品定义与分类口径**（建议至少覆盖 2 类）：
  - **功能完全相同**：面向**同一类用户目标/同一主场景**，提供几乎一致的主流程与关键能力，用户通常会把它们当作“同类产品”来二选一。
    - 判定要点：入口一致（或可互相替换）、主流程一致、关键约束一致（例如是否要求实名/权限、是否依赖同一基础设施）。
    - 示例：
      - 即时通讯 IM：微信/QQ/Facebook Messenger 的消息（message）
      - 代码托管：GitHub/GitLab/Bitbucket（围绕仓库、PR/MR、协作审查的主流程高度相似）
      - 网盘存储：百度网盘/Google Drive/Dropbox（上传-管理-分享-权限的核心链路相近）
    - 分析价值：适合做“**行业共识**”与“**差异化细节**”对照（默认值、引导、容错、风控/权限等）。
  - **核心功能相同**：解决**同一个核心任务/关键子目标**，但载体/平台/形态不同，导致功能边界、使用频率、约束条件存在明显差异。
    - 判定要点：用户要完成的关键任务一致，但“在哪里用/怎么用/能用到什么程度”不同（例如输入方式、网络条件、权限、时效性等）。
    - 示例：
      - 即时通讯：PC 端 QQ vs 手机短信（都能“发消息”，但触达方式、富媒体能力、关系链、费用/限制不同）
      - 记事/知识：手机备忘录 vs Notion/Obsidian（都能“记录与检索”，但协作、结构化、扩展能力不同）
      - 支付：线下 POS 刷卡 vs 手机扫码支付（都能“完成支付”，但成本、链路、风控与场景覆盖不同）
    - 分析价值：适合挖“**关键机制**”而非照搬功能清单（例如更强约束下如何完成同一任务）。
  - **功能本质相同（替代方案/跨行业解法）**：不一定做“同一种产品”，但能满足**同一个底层需求/核心动机**；常用于寻找跨行业的启发式解法与替代路径。
    - 判定要点：用户的核心需求一致（例如“低成本沟通”“获得反馈”“降低不确定性”），但实现方式可能完全不同。
    - 示例：
      - 沟通需求：即时通讯 vs 写信 vs 面对面交流（都满足“传递信息与建立关系”）
      - 获取反馈：应用内问卷/弹窗反馈 vs 客服热线 vs 社群运营（都满足“收集问题与意见”）
      - 学习与辅导：产品新手引导 vs 线下培训/师徒制 vs 交互式教程（都满足“降低上手成本、形成正确操作”）
    - 分析价值：适合在 R3 做方案发散时补足“**非功能性策略**”（流程设计、交互引导、运营机制、服务化替代等）。
- **竞品收集清单（记录来源）**：搜索关键字/同义词、用户访谈、扩展核心功能概念、参考跨行业实现、拆分使用过程关键片段等
- **竞品选择理由**：功能匹配度/知名度（亮点）/用户量/热度（避免“随便找 3 个”）
- **竞品拆解（每个竞品一节，核心产物是细化功能清单）**：
  - 基本信息：产品/版本/平台、目标用户、定价/门槛、证据入口（链接/截图）
  - 选用的拆解方法（至少一种，建议组合使用）：
    - **菜单拆解**：按信息架构/模块把功能罗列出来（适合做完整度扫描）
    - **流程拆解**：按 R1 的主场景走完一条端到端流程，并拆出关键分支（适合看业务链路）
    - **重要操作拆解**：挑“关键交互点/关键决策点”把细节拆到底（适合挖机制与风控）
  - **细化功能清单（必须落盘，建议分层）**：
    - L1：模块/页面/场景（如“创建/提交/审批/查询”）
    - L2：功能点（可被实现/被验收的能力）
    - L3：细节（规则/状态/异常/权限/数据字段/默认值/限制条件）
  - **主流程与关键分支**：用 1 张流程图（Mermaid 优先）呈现“入口→关键步骤→完成条件→中断/回退路径”
  - 关键策略：默认值、引导、容错、风控/权限、审计留痕（如适用）
  - 成本与限制：订阅/实现门槛/学习成本/实施成本/限制条件
  - 可借鉴点 / 明显缺陷（至少：2 条可借鉴 + 1 条缺陷/风险）
- **竞品对比矩阵（用于快速回看，不替代拆解）**：
  - 入口（触发方式）、主流程差异点、关键策略差异点、成本与限制、适用/不适用边界
- **功能/机制整合（面向 R3，输出“功能池”）**：把各竞品的细化功能清单“**去重 + 归类 + 重排**”，沉淀为候选策略/功能池，并为每条功能/机制补充：
  - 来源竞品与证据入口（链接/截图点位）
  - 关联到 R1 痛点/快点/约束的说明（为什么值得带入 R3）
  - 适用前提与关键 trade-off（安全 vs 便利、准确 vs 速度、灵活 vs 一致）
- **结论**：哪些机制是“行业共识”，哪些是“差异化机会”（给出 Top 5，避免泛泛而谈）

### 5.4. 质量门槛（R2-DoD）

- 对比基准绑定到 R1 的场景（任务目标/约束清楚）
- 至少分析 3 个竞品/替代方案，并覆盖至少 2 类竞品口径
- 至少完成一次“流程拆解”或“重要操作拆解”，且有可追溯证据（截图/链接/记录）
- **每个竞品必须输出“细化功能清单”**（建议 L1/L2/L3 分层），并覆盖主流程的关键功能点与细节（规则/状态/异常/权限/默认值等）
- 输出包含“功能/机制整合（功能池）”，可作为 R3 方案细化与 R5 优先级分级的发散输入
- 每个竞品至少给出 2 条“可借鉴机制”与 1 条“明确缺陷/风险”

- **门禁（必须先过）**：必须先输出“候选竞品清单 + 推荐组合”，让用户**多选/补充确认**；在用户确认前不得生成 `competitors.md`

  - 存在“功能/机制整合（功能池）”，可直接供 R3 复用

## 6. 模块 R3：最终方案完善与迭代（基于用户方案输入）

### 6.1. 目标

在完成 R1（`clarify.md`）后，基于：

- **用户输入的解决方案描述**（必须：可以是“完整方案”或“本轮修改点”）
-（可选）R2 竞品/替代方案机制（`competitors.md`：用于机制启发与对照检查）
-（可选）上轮 `solutions.md`（用于多轮迭代）

把“用户方案”稳定加工为**可评审、可落地、可迭代**的 `solutions.md`：结构化表达（机制/流程/规则/状态等）+ 覆盖校验（对齐 clarify）+ 风险登记与验证计划 + 迭代记录。

### 6.2. 输入

**上下文自动获取**：
- 通过上下文自动识别机制获取 `FEATURE_DIR`
- 自动定位 `{FEATURE_DIR}/requirements/clarify.md`
- 自动定位 `{FEATURE_DIR}/requirements/solutions.md`（若存在：作为上轮方案输入，用于迭代）
- 自动定位 `{FEATURE_DIR}/requirements/competitors.md`（若存在：可选输入；用于机制启发与对照检查）

**项目级资源**：
- 项目级约束（技术栈、合规、交付周期、依赖）

**必填用户输入**：
- **解决方案描述**：用户对“要怎么做”的描述（可以是完整方案，也可以是对上轮方案的修改点/新增约束）

### 6.3. 输出（落盘到 `requirements/solutions.md`）

`solutions.md` 的核心职责是回答并可追溯：

1. **用户输入的方案描述**（作为证据入口，可追溯）
2. **当前最终方案是什么**：结构化表达（核心机制/主流程/关键规则/状态/异常/权限/默认值/限制）
3. **它如何覆盖 R1**：对 `clarify.md` 的场景、痛点/快点、边界、约束、干系人关注点逐条对齐（含覆盖表、缺口与应对）
4. **当前方案的优点/缺点**（必须：至少各 3 条，体现权衡；包含对高优先级干系人的影响）
5. **关键风险与关键假设怎么验证**：风险登记 + 验证计划（可执行：Owner/信号/动作明确）
6. **本轮相对上轮改了什么**：迭代记录（多轮追加，不覆盖历史）

### 6.4. 强制门禁与总流程（核心设计思路）

**强制门禁（MUST）**：

- **必须先拿到用户的“解决方案描述”**。在用户提供之前：不得输出/更新 `requirements/solutions.md` 正文（即使是草稿），只能索要用户输入并给出推荐输入格式。
- 全程使用中文；不确定信息必须标注为“未知/待确认”，不得脑补。

**总流程（MUST）**：

1. 获取上下文：`Get-SpecContext` → 得到 `FEATURE_DIR`
2. 读取输入：`clarify.md`（必须）、（可选）上轮 `solutions.md`、（可选）`competitors.md`
3. 门禁提问：请用户提供“完整方案”或“本轮修改点”（推荐格式：A 完整方案 / B 本轮修改点）
4. 结构化与细化：把用户方案描述转为可落盘结构（机制/流程/规则/状态/异常/权限/默认值/限制），并补齐必要细节（不确定则标注“未知/待确认”）
5. 覆盖校验（必须）：逐条对齐 `clarify.md`（场景/痛点/快点/边界/约束/干系人），输出覆盖表与缺口/应对
6. 输出当前方案优点/缺点（必须）：体现权衡，包含对高优先级干系人的影响
7. 落盘：更新/输出 `requirements/solutions.md`（含风险与验证、迭代记录）
8. 自检：按 R3-DoD 检查输出是否达标

### 6.5. 质量门槛（R3-DoD）

- 有“用户输入方案描述”的证据入口（可追溯）
- 当前最终方案已结构化（机制/流程/规则/状态/异常/权限/默认值/限制）
- **覆盖到 `clarify.md`**：对场景、痛点/快点、边界、约束、干系人逐条对齐；缺口已标注并给应对
- 当前方案优点/缺点完整（至少各 3 条），体现权衡
- 风险登记与验证计划可执行（Owner/信号/动作明确）
- 若存在上轮 `solutions.md`：存在追加式迭代记录（不覆盖历史）

### 6.6 执行命令（R3）

- **命令**：`spec-product-solutions`（见 `.aisdlc-cli/commands/spec-product-solutions.md`）
- **门禁（必须先过）**：必须先拿到用户输入的“解决方案描述”（A 完整方案 / B 本轮修改点）；在用户提供前不得生成 `solutions.md`
- **完成后检查**（摘自 `design/product/aisdlc_spec_product_solutions.md`）：
  - 方案已结构化（机制/流程/规则/状态/异常/权限/默认值/限制）
  - 对 `clarify.md` 的场景/痛点/快点/In-Out/约束/干系人关注点有覆盖表，缺口有应对
  - 有优点/缺点（至少各 3 条）且体现权衡
  - 有风险登记与验证计划（Owner/信号/动作明确）
  - 若多轮：有追加式迭代记录（不覆盖历史）

### 6.7 下一步命令（R3 → R5）

## 7. 模块 R5：基于方案生成 PRD

### 7.1. 目标

### 7.2. 输入

**项目级资源**：
- 项目级术语/约束/契约索引（如涉及）

### 7.3. 输出（落盘到 `requirements/prd.md`）

建议最小结构（可直接作为模板）：

- 背景与问题（来自 R1）
- 目标（用户目标/业务目标）与成功指标（可埋点/可观测）
- 范围（In/Out）与里程碑（MVP→迭代）
- 功能列表与优先级：MoSCoW（或 P0/P1/P2），并与里程碑对齐（MVP 至少覆盖 Must/P0）

- 异常与边界（失败、权限、并发、幂等、降级）
- 验收标准（AC）：逐条可测试、可验证
- 关键干系人验收口径：高优先级干系人的关注点如何被满足/如何验收（避免"做完没人认"）
- 风险与依赖（含缓解）
- 追溯链接：引用 `clarify.md`、`solutions.md`、后续 `interaction.md`、`prototype.md`

### 7.4. 质量门槛（R5-DoD）

- 每个核心场景都有 AC（可转测试用例）
- In/Out 与 R1 一致，且与最终方案一致
- 风险与依赖有明确 Owner 与缓解动作
- PRD 的范围/里程碑与功能优先级一致：MVP 至少覆盖 Must/P0，并且 Won't/Out 有明确口径
- 至少覆盖 1 个高优先级干系人的验收口径（能写成可执行的检查项或验收清单）

- **门禁（必须先过）**：必须存在 `requirements/clarify.md` 与 `requirements/solutions.md`；任一缺失不得生成 `prd.md`
- **信息不足时的最小补齐**（摘自 `design/product/aisdlc_spec_product_prd.md`）：
  - 允许最多 **3 个**问题，用于补齐“验收可执行”的必要信息（成功指标/关键规则/验收口径）
  - 用户未回答也要继续输出，但必须在 PRD 中标记“待确认”并说明影响

  - In/Out 与 R1 一致，且与最终方案一致
  - 功能优先级与里程碑一致（MVP 覆盖 Must/P0）

## 9. 模块 R7：基于交互方案生成原型（线框）

### 9.1. 目标

把交互方案变成可评审的"原型资产"，即使不使用 Figma 也能让研发/测试准确理解页面结构与状态。

### 9.2. 输入

**上下文自动获取**：
- 通过上下文自动识别机制获取 `FEATURE_DIR`
- 自动定位 `{FEATURE_DIR}/requirements/interaction.md`

### 9.3. 输出（落盘到 `requirements/prototype.md`）

推荐"文本原型/线框原型说明"结构（可被后续工具再加工为可点击原型）：

- 页面编号与清单（P-001…）
- 每个页面：
  - 页面目的与入口
  - 布局分区（Header/Body/Footer/侧栏等）
  - 信息与控件（字段、按钮、默认值、禁用条件）
  - 状态机（正常/加载/空/错误/权限不足）
  - 跳转与交互（点击/提交/返回/取消/二次确认）
  - 埋点建议（与成功指标对齐，若需要）
- 关键页面线框（可用 ASCII/表格描述，或链接到外部原型文件）
- 最小可用性验证（建议写成固定章节，便于回流）：
  - 测试目标：验证哪些关键假设/痛点是否被解决
  - 测试对象：目标用户（优先）或内部代理（必要时），以及覆盖到的关键干系人视角
  - 任务脚本：按"情景标题→任务目标→成功标准→观察点/记录项"列出
  - 记录方式：问题清单、严重度、复现步骤、建议修复方向
  - 结论与回流：需要回流更新哪些文件（R1/R3/R5/R6），以及为什么

> 若团队使用 Figma：`prototype.md` 仍建议保留"页面清单 + 状态说明 + 关键规则"，并在顶部附上 Figma 链接作为入口，避免原型与规则漂移。

### 9.4. 质量门槛（R7-DoD）

- 页面清单覆盖 R6 的所有节点
- 每个页面至少包含：入口、主要控件、状态、跳转
- 与 PRD 的 AC 可追溯（能指出"哪个页面/哪个状态支持哪条 AC"）
- 至少包含一份"最小可用性验证"章节（含任务脚本与回流指引），确保原型验证能形成闭环而非一次性产物

### 9.5 执行命令（R7）

- **命令**：`spec-product-prototype`（见 `.aisdlc-cli/commands/spec-product-prototype.md`）
- **门禁（必须先过）**：必须存在 `requirements/interaction.md`；缺失不得生成 `prototype.md`
- **关键落盘约束**（摘自 `design/product/aisdlc_spec_product_prototype.md`）：
  - 必须抽取/稳定节点编号（优先复用 R6 的 `P/D/W/T-xxx`；若 R6 未编号则在原型中补齐并建议回流更新 R6）
  - 每页必须写清：入口、主要控件（含规则/默认值/禁用条件）、状态机（正常/加载/空/错/无权限）、跳转与交互、线框描述
  - 若存在 PRD：必须输出 **AC → 页面/状态/验证点**映射表；PRD 缺失则必须标注影响并建议回流补齐
  - 必须包含“最小可用性验证”（任务脚本 + 记录模板 + 回流指引）

### 9.6 下一步（R7 → 评审验证 → 回流闭环）

R7 结束不是终点，建议立刻做一次最小评审闭环：

- **下一步（评审/验证）**：基于 `prototype.md` 组织原型评审与最小可用性验证
- **可选下一步（从线框到可运行）**：若需要把线框快速落到“可运行页面”用于走查对齐，可执行 R8（前端 DEMO 页面生成）
- **发现问题如何回流**（按影响优先）：
  - 口径/边界/场景变化 → 回流 R1（`clarify.md`）
  - 机制/规则/权衡变化 → 回流 R3（`solutions.md`）
  - AC 不可验收/范围冲突 → 回流 R5（`prd.md`）
  - 任务流/节点/状态不合理 → 回流 R6（`interaction.md`）
  - 线框页面结构/内容缺失 → 回流 R7（`prototype.md`）

## 10. 模块 R8：基于线框原型生成前端 DEMO 页面（可选）

> 定位：本步骤属于**概要设计**层面的“快速对齐与走查”，用于把 R7 的线框原型尽快变成可运行的前端页面，方便干系人体验流程与内容。实现细节（路由/目录结构/Mock 数据/组件抽象/生成规则）建议下沉到**详细设计**。

### 10.1. 目标

基于 R7 产出的线框原型（`requirements/prototype.md`），在一个已存在的前端 DEMO 项目中自动生成/更新对应页面与占位内容，使其满足：

- 页面清单与编号对齐（P-001…）
- 关键内容与控件可见（按线框描述生成占位文案/字段）
- 关键跳转可跑通（按线框/交互描述配置路由或导航）
- 明确“这是 DEMO，不是生产实现”（便于管理预期）

### 10.2. 输入

**上下文自动获取**：
- 通过上下文自动识别机制获取 `FEATURE_DIR`
- 自动定位 `{FEATURE_DIR}/requirements/prototype.md`

**用户必填输入**：
- **前端 DEMO 项目根路径**：用于在该项目内生成/更新页面与相关资源（必须是可访问的本地路径）

### 10.3. 输出（概要）

- **Demo 项目代码变更**：新增/更新与线框对应的页面、导航入口与占位内容（可运行、可浏览、可走查）
- **对齐说明落盘**：写入 `{FEATURE_DIR}/implementation/demo.md`（概要级），至少包含：
  - 使用的 DEMO 项目根路径（记录为证据入口）
  - 生成的页面清单（P-xxx → Demo 路由/入口）
  - 已覆盖/未覆盖项与原因（例如：待确认数据口径、缺少权限规则等）

### 10.4. 质量门槛（R8-DoD）

- Demo 页面能启动并访问到主要页面（主场景链路可走通）
- 页面结构与线框一致（至少：布局分区、主要控件、关键文案/占位内容）
- 与 R7 可追溯（能指出“哪个页面/入口对应 prototype 的哪个页面编号”）
- 明确标注 DEMO 属性与边界（避免被误用为生产实现）

## 11. 端到端"最小可用"执行方式（建议）

为了让流程可分布、可迭代，建议采用下面节奏：

- **先做 R1**：澄清完成即可评审（不等竞品/方案）
- **再做 R2/R3**：先宽后窄，快速产出 3 个机制差异明显的方案，并完成决策与验证计划
- **R5 产 PRD**：先产 MVP 版本 PRD（可评审），再迭代补齐细节
- **R6/R7/R8（可选）**：围绕核心场景先出关键链路交互与线框原型；如需可运行走查则执行 R8 生成 DEMO 页面；有重大问题则回流更新 R1/R3/R6/R7，再继续扩展边缘场景
